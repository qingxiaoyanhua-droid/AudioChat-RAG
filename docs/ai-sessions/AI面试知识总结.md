## 一、MHA 是不是 head 分得越多越好？

**结论：不是，需要在「表达能力」和「计算效率」之间做权衡。**

### 核心矛盾

```
head 越多 → 每 head 维度 d_k = d_model / num_heads 越小
           → 每个 head 关注的空间/语义范围越窄
           → 表达能力可能变弱（如果 d_k 太小，单个 head 无法捕捉足够信息）
```

### 三种方案对比

| 方案 | Head 数 | d_k | KV 头数 | 显存（KV Cache） | 表达能力 |
|------|---------|-----|---------|----------------|---------|
| **MHA** | = num_heads | d/n | = num_heads | O(n × L × d) | 强，但显存最大 |
| **MQA** | = num_heads | d/n | **= 1** | O(1 × L × d) | 弱，多头退化 |
| **GQA** | = num_heads | d/n | **= n/g** | O(n/g × L × d) | 平衡，推荐 |

> "GQA 是 MHA 和 MQA 的折中。当 `d_k` 低于 64 时，单个 head 的感受野太小，表达能力急剧下降。所以 Qwen2.5 的设计原则是**保证 d_k >= 64**。比如 72B 模型 d_model=8192，num_heads=64，则 d_k=128，再分到 8 个 KV head，每个 KV head 维度仍然是 128，足够表达语义。"

### 面试加分话术

> "head 数量实际上受两个约束：**（1）d_model 能被 num_heads 整除**；**（2）d_k = d_model / num_heads 需要足够大**（一般 >= 64）。所以在 d_model 固定的情况下，head 数量有一个合理上限。比如 BERT-Large (d=1024) 用 16 个 head，d_k=64，正好是临界值。超过这个临界，d_k 太小，attention 的表达能力反而退化。而且从硬件层面看，head 太多会导致 `query_states` 矩阵变宽，反而不利于并行计算。"

### OpenCLIP 训练核心思想

> "OpenCLIP 的核心是 **Contrastive Learning（对比学习）**。给定 N 个 (image, text) 对组成的 batch，图像编码器和文本编码器分别输出 N 个 embedding。然后算 N×N 的 cosine similarity 矩阵。对角线上的 N 个是正确的 image-text pair，应该让相似度接近 1；非对角线是负样本，应该让相似度接近 0。用 InfoNCE / NT-Xent loss 同时从 image→text 和 text→image 两个方向优化。"

## 四、为什么选 YOLO11？优势在哪？

### 面试回答模板（建议分 3 层说）

**第一层 — 为什么不用 v8 / v9 / v10：**

> "YOLOv8 是 Ultralytics 的上一代旗舰，解耦头和 anchor-free 设计已经很成熟。但 v8 的 C2f 模块参数量偏大，在边缘设备上效率一般。YOLOv9 和 v10 虽然各有创新（PGI 梯度信息、NMS-Free），但 v9 的 GELAN 架构复杂，v10 虽然去掉了 NMS，但当时生态不够成熟，部署工具链没有跟进。而 YOLO11 是 Ultralytics 官方在 v8 基础上**做减法**的版本——用更少的参数达到更好的精度，同时保持完整的 Ultralytics 生态（训练/部署/导出无缝衔接）。"

**第二层 — YOLO11 的核心优势（3 点）：**

1. **C3k2 瓶颈**：比 C2f 更紧凑的 CSP 变体，`kernel_size=3` 的小卷积替代大卷积，减少 30% 参数量同时保持感受野
2. **C2PSA 注意力**：CSP + 并行空间注意力，以极低代价增强特征聚焦，对小物体检测帮助明显（mAP↑2~3%）
3. **多任务统一**：同时支持检测、分割、pose、OBB 一个框架搞定，不需要像 v8 那样切换多个模型

**第三层 — 实际项目中的选择理由（结合你的项目）：**

> "在实际项目中，我们用 YOLO11 做的是会议场景下的**说话人检测/手势检测**，对推理延迟要求高（实时流式处理）。YOLO11n 在 CPU 上只需要 56ms，比 v8n 快 30%，精度还高 2%。加上 Ultralytics 的 Python API 和 ONNX 导出非常方便，我们直接把模型嵌入到流式 pipeline 里，作为前置的事件检测模块。"

## 五、PageAttention V2 相对于 V1 做了什么改进？

**本质区别：V1 基于标准 FlashAttention-2，V2 基于 FlashAttention-3 + Flash Decoding + Paged KV Cache**

```
┌─────────────────────────────────────────────────────────────────┐
│                    PageAttention v1                              │
│  (基于 FlashAttention-2)                                         │
│                                                                 │
│  问题：                                                           │
│  ├── Prefill 阶段：对整个序列做分块矩阵乘法，效率高                  │
│  └── Decode 阶段：每次只生成 1 个 token，仍需加载完整 KV Cache       │
│       → GPU 利用率极低（< 10%），大量时间在等显存读取               │
│                                                                 │
│  V1 的 KV Cache 是连续的（contiguous），需要预分配连续显存          │
│  → 显存碎片化问题：实际可用显存 << 理论显存                         │
└─────────────────────────────────────────────────────────────────┘

                         ▼ 改进

┌─────────────────────────────────────────────────────────────────┐
│                    PageAttention v2                              │
│  (基于 FlashAttention-3 + Paged KV Cache)                       │
│                                                                 │
│  改进 1: Paged KV Cache                                          │
│  ├── 灵感来自操作系统内存分页管理（OS virtual memory）             │
│  ├── KV Cache 不再预分配连续显存，而是按 "页" (block) 动态分配      │
│  ├── 相同 block_size=16，物理显存可以是不连续的                    │
│  ├── 显存利用率从 ~60-70% 提升到 ~90%+                            │
│  └── 吞吐提升 2-4 倍                                             │
│                                                                 │
│  改进 2: FlashAttention-3 (Hopper 架构)                          │
│  ├── FP8 计算支持 (H100+)，Tensor Core 利用率更高                 │
│  ├── WGMMA (Warp-group Matrix Multiply Accumulate)               │
│  ├── 分页预取 + 异步执行，隐藏内存延迟                             │
│  └── 长序列场景下 FWD 速度比 FA-2 快 1.5~2x                       │
│                                                                 │
│  改进 3: Flash Decoding（解码专项优化）                           │
│  ├── Prefill: batch 级别的 FlashAttention                        │
│  └── Decode: 每步只处理 1 个新 token，但利用 KV Cache 分页          │
│       → 即使是自回归 decode，GPU 利用率也大幅提升                  │
└─────────────────────────────────────────────────────────────────┘
```

### 核心数值对比

| 指标 | PageAttention V1 | PageAttention V2 |
|------|----------------|-----------------|
| KV Cache 管理 | 连续分配，预分配 | 分页管理，动态分配 |
| 显存利用率 | ~60-70% | ~90%+ |
| Decode 吞吐 | 基准 | **2-4x 提升** |
| 长序列 Prefill | 正常 | **1.5-2x 提升** |
| 硬件支持 | Ampere+ | **Hopper+ (H100)** + 兼容 Ampere |
| 量化支持 | FP16/BF16 | **FP8 + FP16/BF16** |

> "V2 的本质改进是：**让显存管理从'预分配连续块'变成'按需分页'**。打个比方，V1 像住酒店一次性订 100 间连续房间，如果中间有人退房，空出来的房间没法被复用；V2 像 Airbnb 按需订单间，退房后房间立即可被新客使用。所以 V2 的并发数（throughput）大幅提升，特别适合 vLLM 这种多请求并发的推理引擎。"

> "BGE-large-zh-v1.5 的核心定位是**中文稠密检索（BGE = BAAI General Embedding）**。它用 24 层 BERT-style Transformer，输出 1024 维向量。v1.5 相比 v1 的最大改进是**相似度分布校准**——v1 的 cosine similarity 全集中在 [0.6, 1.0]，区分度差；v1.5 通过对比学习温度调整，让相似度分布更合理，不用 instruction 也能保持好的检索效果。典型应用场景是 RAG 检索 + 语义 reward 信号计算。"
