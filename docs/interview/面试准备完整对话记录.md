### GRPO 奖励函数（五维度）
```python
reward = 0.40×准确性 + 0.25×流畅度 + 0.15×相关性 + 0.10×结构化 + 0.10×长度
```

| 维度 | 权重 | 计算方法 |
|------|------|----------|
| **准确性** | 40% | BGE Embedding + 余弦相似度（vs reference） |
| **流畅度** | 25% | N-gram 重复度（重复越少越好） |
| **相关性** | 15% | BGE Embedding + 余弦相似度（vs query） |
| **结构化** | 10% | Markdown 格式检查（标题/列表/加粗） |
| **长度** | 10% | 是否在 100-300 字范围 |

## 5. RAG 检索增强

### 存储流程
```
会议记录 → 分块（按说话人） → BGE Embedding → ChromaDB
```

### 检索流程
```
Query → 语义检索 → 时间衰减重排序 → Top-K
```

### 时间衰减公式（核心）
```python
def time_decay(timestamp, half_life_days=7.0):
    doc_time = datetime.fromisoformat(timestamp)
    now = datetime.now()
    days_diff = (now - doc_time).days
    decay = 0.5 ** (days_diff / half_life_days)
    return max(0.1, decay)  # 最小 0.1
```

**例子**：
- 今天：decay = 1.0
- 7 天前：decay = 0.5
- 14 天前：decay = 0.25
- 30 天前：decay = 0.1

## 6. 时间衰减优化方案

### 面试官问题
> "如果用户需要久远但重要的信息（如架构决策），但近期没有记录，时间衰减会让权重很低，怎么解决？"

### 4 层解决方案

| 方案 | 实现难度 | 效果 | 状态 |
|------|----------|------|------|
| **1. 关键词豁免** | ⭐ | 中 | ✅ 已实现 |
| 2. 意图识别 | ⭐⭐ | 中 | 🔄 方案中 |
| 3. 混合检索 | ⭐⭐ | 高 | ⭐ 推荐 |
| 4. 元数据过滤 | ⭐⭐⭐ | 高 | ⭐ 推荐 |
| **5. 父块检索** | ⭐⭐ | 高 | ⭐ 新增 |

### 方案 1：关键词豁免（已实现）

```python
TIMELESS_KEYWORDS = ["架构", "决策", "负责人", "决定", "核心"]

def retrieve(query, k=3):
    is_timeless = any(kw in query for kw in TIMELESS_KEYWORDS)
    
    docs = storage.search(query, k=k*2)
    for doc in docs:
        relevance = compute_similarity(query, doc.content)
        decay = 1.0 if is_timeless else time_decay(doc.timestamp)
        doc.score = relevance * decay
    
    docs.sort(key=lambda x: x.score, reverse=True)
    return docs[:k]
```

### 方案 3：混合检索（推荐）

```python
def hybrid_retrieve(query, k=3):
    # 第一路：语义检索 + 时间衰减（查近期）
    semantic_docs = storage.search(query, k=k)
    for doc in semantic_docs:
        doc.score = similarity * time_decay(doc.timestamp)
    
    # 第二路：关键词检索（查历史）
    keyword_docs = storage.keyword_search(query, k=k)
    for doc in keyword_docs:
        doc.score = doc.keyword_score  # 不受时间影响
    
    # 合并两路结果
    all_docs = merge(semantic_docs, keyword_docs)
    all_docs.sort(key=lambda x: x.score, reverse=True)
    return all_docs[:k]
```

### 方案 5：父块检索（新增）

**问题**：
- 小块信息不完整（行动项和负责人可能分开）
- 同一父块的多个小块会重复返回

**解决方案**：
```python
def retrieve_with_parent_chunks(query, k=3):
    # 1. 语义检索小块（粗排）
    small_chunks = storage.search(query, k=50)  # 返回 50 个小块候选
    
    # 2. 按父块分组（合并同父块的小块）
    parent_groups = {}
    for chunk in small_chunks:
        parent_id = chunk["parent_id"]
        if parent_id not in parent_groups:
            parent_groups[parent_id] = {
                "parent": get_parent_chunk(parent_id),  # 获取完整父块
                "child_chunks": [],
                "max_similarity": 0,
            }
        parent_groups[parent_id]["child_chunks"].append(chunk)
        parent_groups[parent_id]["max_similarity"] = max(
            parent_groups[parent_id]["max_similarity"],
            chunk["similarity"]
        )
    
    # 3. 计算父块分数（时间衰减 + 最高相似度）
    scored_parents = []
    for parent_data in parent_groups.values():
        parent = parent_data["parent"]
        decay = time_decay(parent["timestamp"])
        final_score = parent_data["max_similarity"] * decay
        
        scored_parents.append({
            "parent": parent,
            "score": final_score,
            "child_count": len(parent_data["child_chunks"]),
        })
    
    # 4. 竞拍选 Top-K（按分数排序）
    scored_parents.sort(key=lambda x: x["score"], reverse=True)
    return [item["parent"] for item in scored_parents[:k]]
```

**优势**：
- 信息完整：行动项 + 负责人在一起
- 避免重复：同一父块的多个小块合并
- 更准确：父块语义更完整

**Action（行动）**：

> 2. SFT+GRPO 微调千问 0.5B，五维度奖励函数
> 3. RAG 检索增强：ChromaDB + BGE 语义检索 + 时间衰减重排序
> 4. 流式处理：500ms 切片，边说话边转录

**Result（结果）**：
> - 转写准确率：92%
> - Top-3 检索准确率：83%
> - RAG 提升问答准确性：31%（0.64 → 0.84）
> - GRPO 提升生成质量：15%（0.68 → 0.78）
> - 会议处理时长缩短：30%

---

### 高频问题及回答

#### Q1: "为什么用 GRPO 不用 PPO？"
> "PPO 需要 Critic 模型，实现复杂。GRPO 用组内相对优势代替，无需 Critic，实现简单。实验中 GRPO 和 PPO 效果接近，但训练时间少 30%。"

**核心公式**：
```python
# 组内相对优势
A_i = (r_i - mean(r)) / std(r)

# GRPO 损失
L = -E[log(π_new/π_old) × A] + kl_coef × KL
```

---

#### Q2: "奖励函数怎么设计的？"
> "五维度加权：0.40×准确性 + 0.25×流畅度 + 0.15×相关性 + 0.10×结构化 + 0.10×长度。准确性用 BGE Embedding+ 余弦相似度，流畅度用 N-gram 重复度。"

---

#### Q3: "RAG 检索不到相关内容怎么办？"
> "有 4 层解决方案：1.关键词豁免（已实现）2.意图识别 3.混合检索 4.元数据过滤。核心思想是时间衰减不能牺牲历史重要信息。"

---

#### Q4: "数据怎么来的？"
> "SFT 和 GRPO 数据是规则生成的，用于验证方案。测试集人工标注了 50 条，确保评估准确。如果落地，会用千问蒸馏 + 人工抽检。"

---

#### Q5: "为什么用 0.5B 不用 7B？"
> "0.5B 成本低（单卡 3090）、速度快（SFT 30 分钟）、易部署（8GB 显存）。会议总结不需要复杂推理，0.5B 微调后足够用。"

---

#### Q6: "时间衰减的缺陷怎么解决？"
> "有 4 层解决方案：
> 1. 关键词豁免：核心术语（架构、决策）不打折
> 2. 意图识别：溯源类问题关闭时间衰减
> 3. 混合检索：语义 + 关键词两路合并
> 4. 父块检索：小块粗排 + 父块精排，信息更完整
>
> 目前实现了关键词豁免，后续计划做混合检索。"
