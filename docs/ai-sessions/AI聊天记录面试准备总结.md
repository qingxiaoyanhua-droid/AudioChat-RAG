### 自适应 KL 系数实现

```python
# 每 step 结束后
kl = compute_kl(pi_new, pi_ref)  # 当前 KL 散度
if kl > 0.2:       # 跑太远
    beta *= 1.5    # 加大惩罚
elif kl < 0.05:     # 太保守
    beta *= 0.5    # 减小惩罚，允许探索
```

## 四、GRPO 公式推导

```
1. 采样：对 prompt x，从 π_θ 采 G 个回复 {y₁...y_G}
2. 打分 + 归一化（组内相对优势）：
   r_i = reward(x, y_i)
   A_i = (r_i - mean(r)) / std(r)
3. 重要性采样比率：
   ratio_i = π_θ(y_i|x) / π_old(y_i|x)
4. PPO-clip：
   L_clip = min(ratio × A, clip(ratio, 1-ε, 1+ε) × A)
5. 最终 loss：
   L = -E[L_clip] + β × KL(π_θ || π_ref)
```

**GRPO 和 PPO 的核心区别：**
- PPO：Actor-Critic，需要 Critic 估计 baseline V(s)
- GRPO：用组内均值代替 Critic，不需要 value network，省显存

**GRPO 和 DPO 的核心区别：**
- DPO：offline + off-policy，事先收集偏好对，不需要 reward model
- GRPO：online + on-policy，实时采样，不需要偏好标注

## 五、6 维奖励函数

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| 准确性 | 0.30 | BGE sim(生成答案, 标准答案) |
| 忠实度 | 0.20 | 逐句 vs RAG Top3，BGE sim<0.5 算幻觉，1-幻觉率 |
| 流畅度 | 0.15 | distinct-4 n-gram 去重率 = unique_4gram / total_4gram |
| 相关性 | 0.15 | BGE sim(生成答案, 用户 query) |
| 结构化 | 0.15 | 正则检查：有 ## 标题、行动项、决策项、列表格式 |
| 长度 | 0.05 | 目标 200-500 字，太短/太长扣分 |

**流畅度实现（n-gram 重复惩罚）：**

```python
def fluency_score(text):
    tokens = list(text)
    ngrams = [tuple(tokens[i:i+4]) for i in range(len(tokens)-3)]
    unique_ratio = len(set(ngrams)) / len(ngrams)  # 1.0 不重复, 0.1 严重复读
    return unique_ratio
```

**幻觉率计算：**

```python
# 逐句检查，每句和 RAG Top3 文档算 BGE sim
# max_sim < 0.5 → 这句是幻觉
# 幻觉率 = 有幻觉的回答数 / 总回答数
```

**权重怎么定的：**
1. 任务优先级：准确+忠实最重要，各 0.30 和 0.20
2. 消融验证：逐个去掉某维度看效果崩不崩
3. 关键维度微调：准确性试了 0.25/0.30/0.35，0.30 时 reward 最高且幻觉率最低

## 六、RAG 两阶段检索架构

### 完整链路（必须能画出来）

```
Stage 1: Bi-Encoder 粗排
  query → BGE Embedding 编码 → ChromaDB 余弦相似度 → Top-15
  （纯语义排序，不掺时间因素）

Stage 2: Cross-Encoder 精排
  (query, doc) 拼接 → BGE Reranker 交叉注意力 → 分数

Stage 3: Soft Decay 排序
  final_score = rerank_score × √(decay)
  半衰期 7 天，decay = 0.5^(days/7)

Stage 4: 最终输出
  Top-3 → 拼入 RAG Prompt → LLM 生成
```

### 为什么用 sqrt(decay) 而不是 decay

```
30 天前: decay = 0.05
原版: rerank 0.9 × 0.05 = 0.045 → 基本淘汰
√版: rerank 0.9 × √0.05 = 0.9 × 0.22 = 0.20 → 仍有竞争力
等效半衰期从 7 天拉长到 14 天
```

### 关键词豁免机制

架构/决策类关键词（"架构选型"、"技术方案"、"决策"）不参与时间衰减，确保关键内容不被稀释。

### 评测方法

```
Top-3 准确率 85%：
  - 200 条测试 query，按时间分层抽样
  - 人工标注 ground truth 会议 ID
  - Top-3 中有来自正确会议的 chunk → 命中
  - 170/200 = 85%

问答准确性提升 31%：
  - 200 条 query，无 RAG 生成答案 vs 标准答案，BGE sim = 0.64
  - 有 RAG 生成答案 vs 标准答案，BGE sim = 0.84
  - (0.84-0.64)/0.64 = 31%
```

## 七、Badcase 归因与解决（面试必问）

### Badcase 1：重复关键句刷分

- **现象**：`后端API完成80%。后端API已完成80%。后端API进度80%`
- **归因**：只有准确性和流畅度两个维度，重复关键句能同时拿两个高分
- **解决**：加结构化奖励（检查 Markdown 格式）+ 流畅度 n-gram 惩罚
- **效果**：复读率从 23% 降到 5%

### Badcase 2：编造不存在的数字（幻觉）

- **现象**：会议没讨论预算，生成"预算50万元"
- **归因**：LLM 倾向于生成具体数字来拿准确性高分
- **解决**：加忠实度维度，逐句和 RAG 文档比对
- **效果**：幻觉率从 28% 降到 18%

### Badcase 3：时间混淆（问新内容返回旧会议）

- **现象**：问"本周进度"，返回上个月会议
- **归因**：Bi-Encoder 语义上"项目进度"相似，分不出时间
- **解决**：加时间衰减 + Soft Decay
- **效果**：时间类 query 准确率从 71% 提到 84%

### Badcase 4：训练中期 reward 崩塌

- **现象**：step 400 左右，reward 从 0.65 突然掉到 0.3
- **归因**：某 batch 梯度太大，ratio 最大值到 8.0（正常 0.8-1.2）
- **解决**：确认 clip ε=0.2 生效 + lr 从 1e-5 降到 1e-6
- **效果**：后续训练稳定收敛

## 八、分布式训练（可能被追问）

### DDP vs DP

```
DP（旧）：所有梯度汇总到 GPU 0 再分发，通信不均匀，有主卡瓶颈
DDP（现在标准）：AllReduce 环形通信，每卡收发量相等，无主卡瓶颈
```

### ZeRO 三个阶段

```
ZeRO-1：切分优化器状态（Adam m,v 占显存大头），省约 4×
ZeRO-2：再切分梯度，用 Reduce-Scatter
ZeRO-3：连参数也切分，前向/反向时 AllGather 临时拿参数

通信量：ZeRO-3 > ZeRO-2 > ZeRO-1 > 无 ZeRO
显存节省：ZeRO-3 > ZeRO-2 > ZeRO-1 > 无 ZeRO
```

**AllReduce / ReduceScatter / AllGather 区别：**
```
AllReduce：所有卡数据求和，结果每卡都有完整副本（DDP 梯度同步）
ReduceScatter：求和后每卡只留 1/N（ZeRO-2）
AllGather：每卡有一块，收集成完整的（ZeRO-3 参数拼回）
AllReduce = ReduceScatter + AllGather
```

**为什么你的 0.5B 模型不需要 ZeRO：** 单卡 40GB 足够放下，不需要分片。

### LoRA 参数

```
rank = 32, alpha = 16
实际缩放比 = alpha/rank = 0.5
可训练参数 = 2 × rank × d + rank + rank ≈ 1.6M（占 0.3%）
```


## 十一、SFT 和 GRPO 数据关系（易被追问）

### 常见误解澄清

```
❌ 错误理解：GRPO 用的 prompt 不应该和 SFT 一样，否则是数据泄漏
✓ 正确理解：SFT 和 GRPO 用同一批训练数据是标准做法（DeepSeek R1 等都这么干）
```

### 正确逻辑

```
SFT：模型看到 (输入, 标准答案)，学会输出格式（交叉熵 loss）
GRPO：模型只看到 (输入)，自己采样回复，奖励函数打分
     优化的是"生成策略"，不是"复述标准答案"
```

### 为什么 GRPO 仍然有效

- SFT 学"说什么"，GRPO 学"怎么说得更好"
- SFT 后准确性已经很高，GRPO 核心贡献在忠实度（降幻觉）
- 消融验证：去掉 GRPO，准确性只降 2%，但幻觉率升 10 个点

### 什么才是真正的数据泄漏

```
训练集 250 场 → SFT + GRPO 训练 ✅ 没问题
测试集 50 场 → 只用来算最终指标 ✅ 完全隔离
拿测试集做 GRPO → ❌ 数据泄漏
```

### RAG 知识库存的是什么

```
RAG 知识库：会议原始转录文本（按发言切分的 chunk）
  "[00:05] 李四：后端要重构"
  "[00:12] 张三：那就微服务吧"
  → 原始发言，不是总结

SFT 标准答案：72B 生成的会议纪要总结
  "## 会议纪要\n- 决定采用微服务架构重构后端"
  → 总结，没有存进 RAG
```

## 十二、面试高频追问链

### GRPO 追问链

1. "GRPO 和 PPO 区别？" → 用组内均值代替 Critic
2. "优势 A_i 怎么算？" → A_i = (r_i - mean) / std
3. "为什么组内均值可以做 baseline？" → 不需要额外训练 Critic
4. "group_size 太小方差大怎么办？" → 设置为 4 已验证足够

### RAG 追问链

1. "两阶段检索为什么粗排 15 个？" → Cross-Encoder 复杂度 O(n)，15 对约 120ms
2. "Soft Decay 为什么用 sqrt？" → 等效半衰期翻倍，让时间做微调不主导
3. "时间衰减在哪个阶段施加？" → 最终打分时乘，不在精排前

### 奖励函数追问链

1. "6 维权重怎么定的？" → 业务优先级 + 消融验证 + 手动微调
2. "为什么忠实度权重 0.20 不是 0.10？" → 试过 0.10 幻觉率 26%，不够
3. "做过 grid search 吗？" → 没资源，跑一组要 2 小时

## 十五、项目不足与改进方向（面试被问"有什么不足"）

> 建议从这三个方向回答，展示你知道下一步怎么做：

1. **奖励函数是手写规则，不是 learned RM**
   - 好处：迭代快、可解释、不需要额外标注
   - 改进方向：用偏好数据训一个 Bradley-Terry RM

2. **测试集偏小**（200 条）
   - 改进方向：扩到 500+ 并加人工评测

3. **Chunk 粒度细**（单句切分缺上下文）
   - 改进方向：滑动窗口语义分块（相邻 3-5 句重叠 1-2 句）
