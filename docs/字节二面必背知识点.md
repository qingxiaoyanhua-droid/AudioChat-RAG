### 1. Kafka 消息可靠性三机制

**面试官原问**：Kafka 内部是怎么保证消息不丢的？

**回答框架**：从生产者 → Broker → 消费者三个环节说，每个环节各一道保险。

#### 环节一：生产者（Producer）

```
问题：生产者发出去的消息在半路丢了怎么办？

答案：配置 acks=all + retries 自动重试

acks=0：发出去就不管了，可能丢
acks=1：leader 副本写入成功就返回，可能丢（follower 还没同步完就挂了）
acks=all：所有 ISR 副本（in-sync replica）全部写入成功才返回，最可靠

retries=MAX：发送失败自动重试，直到成功
```

#### 环节二：Broker（服务端）

```
问题：Broker 挂了，消息还在吗？

答案：副本机制 + 持久化

副本因子 replication.factor >= 3：每条消息存 3 份
min.insync.replicas >= 2：写入时至少 2 个副本确认才算成功
刷盘策略：log.flush.interval.ms 控制多久刷一次盘（默认由 OS 决定，OS 满了才刷）

ISR 是什么：
  - ISR = In-Sync Replicas，跟上 leader 进度的副本集合
  - 如果 follower 同步太慢（lag 太大），会被踢出 ISR
  - 只有 ISR 里的副本才能参与选举新 leader
```

#### 环节三：消费者（Consumer）

```
问题：消费者处理到一半崩了，重启后从哪开始消费？

答案：手动提交 offset，不自动提交

enable.auto.commit=false
处理完业务逻辑 → 手动调用 consumer.commitSync()
（注意：先处理业务，再提交 offset，否则处理失败但 offset 已提交会丢消息）

消费者的两个位置概念：
  - committed offset：已提交的消费进度
  - current offset：当前消费到哪了
```

#### 一句话总结

> "Kafka 保证消息不丢靠三道保险：生产者 acks=all 确认所有副本收到；Broker 用 ISR 副本机制 + 持久化；消费者手动提交 offset，处理完再提交。"

### 2. Python 装饰器原理

**面试官原问**：装饰器的原理是什么？

**核心**：装饰器 = 高阶函数 + 闭包。两句话讲清楚。

#### 原理拆解

```python
# 装饰器的本质：接受一个函数，返回一个新函数
def my_decorator(func):          # 1. 高阶函数：接受函数作为参数
    def wrapper(*args, **kwargs):  # 2. 闭包：wrapper 能访问 func 的变量
        print("调用前增强")
        result = func(*args, **kwargs)  # 3. 调用原函数
        print("调用后增强")
        return result
    return wrapper              # 4. 返回新函数

@my_decorator          # 等价于：say_hello = my_decorator(say_hello)
def say_hello(name):
    print(f"Hello {name}")
```

#### 装饰器的三个应用场景（必背）

| 场景 | 代码 | 作用 |
|------|------|------|
| 日志记录 | `@log` | 函数调用前后打印入参出参 |
| 缓存 | `@lru_cache(maxsize=128)` | 相同入参直接返回缓存结果 |
| 重试 | `@retry(max_attempts=3)` | 失败自动重试 N 次 |
| 权限校验 | `@require_login` | 调用前检查登录状态 |
| 性能计时 | `@timeit` | 统计函数执行时间 |

#### 带参数的装饰器

```python
# 带参数装饰器需要多一层嵌套
def retry(max_attempts=3):
    def decorator(func):
        def wrapper(*args, **kwargs):
            for i in range(max_attempts):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if i == max_attempts - 1:
                        raise
        return wrapper
    return decorator

@retry(max_attempts=5)
def call_api():
    pass
```

#### 一句话回答

> "装饰器本质上是一个高阶函数，它接受原函数作为参数，返回一个新函数，在新函数里可以在调用前后插入增强逻辑。Python 的 `@decorator` 语法糖等价于 `func = decorator(func)`。"

### 3. ASR-声纹分离-SR 三个概念分清

**面试官追问**：你做了说话人分离，说话人识别用的什么模型？训练的哪一块？

**你暴露的问题**：把 ASR（语音识别）、SD（说话人分离）、SR（说话人识别）三个东西混在一起说了，面试官多次追问想帮你理清，但你越答越乱。

#### 三个概念对照

| 缩写 | 全称 | 任务 | 输入 | 输出 | 你的项目里用的模型 |
|------|------|------|------|------|-----------------|
| **ASR** | Automatic Speech Recognition | 把语音转成文字 | 音频（任意说话人） | "今天天气很好" | FunASR Paraformer |
| **SD** | Speaker Diarization（说话人分离） | 找出几个人在说话，各自说了哪些段落 | 音频 | 说话人1:[0-5s], 说话人2:[5-10s]... | FunASR 内置 / 3D-Speaker |
| **SR** | Speaker Recognition（说话人识别） | 判断这段音频是谁说的（1:1 或 1:N） | 音频片段 + 注册音频 | 相似度分数 / 用户ID | **3D-Speaker（你实际用的是这个，不是 FunASR）** |

#### 关键区分

```
你的项目实际链路：
  原始音频
    → ASR（FunASR）：转成文字
    → SD（3D-Speaker diarization）：分离出几个人、每段谁说的
    → SR（3D-Speaker embedding）：把说话人片段 embedding 与注册用户的 embedding 比对，判断是谁

常见错误：
  ❌ "FunASR 做了说话人识别"  → FunASR 做的是 ASR，不是 SR
  ✅ "3D-Speaker 做了说话人分离和识别"
```

#### SR 的两种模式

```
1:1 验证（Speaker Verification）：
  问题："这两个人是不是同一个人？"
  场景：声纹解锁、支付验证
  输出：相似度分数 > 阈值 → 同一个人

1:N 识别（Speaker Identification）：
  问题："这段音频是谁说的？"
  场景：会议记录标注说话人
  输出：在候选集里找最相似的那个
```

#### 一句话回答

> "ASR 把语音转文字，SD 把音频切分成不同说话人的段落，SR 负责判断每段音频具体是谁说的。我的项目里 FunASR 做 ASR，3D-Speaker 做 SD 和 SR。"

### 4. 说话人识别评测指标

**面试官原问**：你们怎么评测说话人识别？只有准确率一个指标吗？

**你暴露的问题**：只答了准确率，没说清楚 FAR/FRR/EER 这些关键指标。面试官直接指出：**"你光看准确率不行，准确率和召回率是 trade-off，必须看完整套评测体系。"**

#### 说话人验证（1:1）指标

```
场景：判断"这两个人是不是同一个人"

FAR（False Accept Rate）误接受率：
  - 定义：不该通过但通过了
  - 公式：FAR = 假阳性 / (假阳性 + 真阴性)
  - 影响：越低越安全（门禁场景 FAR 高会很糟糕）

FRR（False Reject Rate）误拒绝率：
  - 定义：该通过但没通过
  - 公式：FRR = 假阴性 / (假阴性 + 真阳性)
  - 影响：越低用户体验越好

EER（Equal Error Rate）：
  - 定义：FAR = FRR 那个点
  - 意义：EER 越低，模型越好
  - 为什么用它：它是单数字，方便比较不同模型
```

#### FAR 和 FRR 的关系

```
阈值设高 → FAR 低（严格，难通过），FRR 高（误杀好人）
阈值设低 → FAR 高（容易误放），FRR 低（好人放行多）

这俩是 trade-off：
  提高准确率 → 召回率下降
  提高召回率 → 误报率上升
  → 没有同时最优的阈值，所以用 EER 这个交点来评估模型整体水平
```

#### 说话人识别（1:N）指标

```
Top-1 准确率：最相似的就是正确答案的概率
Top-5 准确率：相似度前5里有正确答案的概率
假阳性率（FP Rate）：把 A 错认成 B 的比例
假阴性率（FN Rate）：该认出来但没认出来的比例
mAP（平均精度）：综合考虑排序质量和识别准确率
```

#### 说话人分离的专属指标

```
DER（Diarization Error Rate）：
  - 说话人分离错误率，最核心指标
  - DER = (假阳性时间 + 假阴性时间 + 混淆时间) / 总时长
  - 理想值：DER 越接近 0 越好

JER（Jaccard Error Rate）：
  - DER 的宽松版本，用 Jaccard 相似度计算，允许时间对齐有偏差
  - 通常 JER >= DER

WER（Word Error Rate）：
  - ASR 转录的文字准确率
  - WER = (插入 + 删除 + 替换) / 总词数
  - 越低越好，ASR 场景用这个
```

#### 完整评测体系

```
你的项目应该关注：

说话人分离（SD）：
  → DER、JER（看分得准不准）

说话人识别（SR）：
  → EER（1:1 验证）
  → Top-1/Top-5 准确率（1:N 识别）
  → FAR、FRR（评估安全性）

ASR 转录质量：
  → WER（文字准确率）

二面如果问到，直接说：
> "我们端到端评测不仅看准确率，还看 DER（说话人分离错误率）和 EER（说话人验证的等错误率），以及 FAR/FRR 的 trade-off 曲线。"

### 5. 进程-线程-协程体系

**面试官追问**：不同机器规格并行方式不一样，核数高和核数低怎么做？

**你暴露的问题**：回答涉及了 GPU 通信、张量并行，但被追问单核怎么办时开始乱。

#### 体系框架：按场景选择

```
核心问题：Python 里 GIL（全局解释器锁）限制了同一时刻只有一个线程执行 Python 字节码

所以：
  CPU 密集型任务 → 多进程（绕过 GIL）
  IO 密集型任务 → 多线程 或 协程
  混合型 → 看情况
```

#### 单核 / 低配置机器：协程（asyncio）

```python
import asyncio

async def fetch_user():
    # 协程：遇到 IO 自动切换，不阻塞
    result = await http_get("/user/1")
    return result

async def main():
    # 单线程内并发处理大量 IO 任务
    tasks = [fetch_user() for _ in range(1000)]
    results = await asyncio.gather(*tasks)

# 原理：事件循环驱动，协程遇到 await 就让出 CPU
# 优点：开销极低（一个协程只占用几 KB）
# 缺点：只能处理 IO 密集，不能绑住 CPU
```

#### 多核 CPU 密集型：多进程（multiprocessing）

```python
from multiprocessing import Pool

def cpu_task(x):
    # 每个进程有独立的 GIL，真正并行
    return heavy_compute(x)

with Pool(8) as p:   # 8 个进程
    results = p.map(cpu_task, range(1000))

# 优点：绕过 GIL，真正并行
# 缺点：进程间通信（IPC）开销大，数据共享麻烦
```

#### 多核 IO 密集型：多线程

```python
from concurrent.futures import ThreadPoolExecutor

def io_task(url):
    # 线程池：IO 等待期间 GIL 被其他线程释放
    return http_get(url)

with ThreadPoolExecutor(max_workers=100) as executor:
    futures = [executor.submit(io_task, url) for url in urls]
    results = [f.result() for f in futures]

# 注意：Python 里线程在 IO 等待时会释放 GIL
# 所以 IO 密集场景多线程仍然有效
# CPU 密集场景多线程无效（GIL 导致串行）
```

#### 三者对比

| | 协程 | 多线程 | 多进程 |
|---|---|---|---|
| 执行单元 | 单线程内的函数 | 线程（OS 调度） | 进程（独立地址空间） |
| GIL | 不涉及 | IO 时自动释放，CPU 时阻塞 | 不影响（各有各的 GIL） |
| 通信开销 | 极低（共享内存） | 中等（共享内存 + 锁） | 高（需要 IPC） |
| 适合场景 | 高并发 IO | 中并发 IO | CPU 密集 |
| 启动开销 | 极小（KB 级） | 小 | 大（MB 级） |
| 并行度 | 单核（假并发） | 多核 | 多核（真并行） |

#### 一句话回答

> "单核低配机器做 IO 密集用协程（asyncio），单核 CPU 密集没有好办法；多核 CPU 密集用多进程绕过 GIL，多核 IO 密集用多线程，协程始终是单线程只是开销极低。"

### 6. 微服务通信方式

**面试官原问**：微服务之间有哪些通信方式？

**你暴露的问题**：只答出了 MQ 生产消费和 RPC 两个词，说得不完整。

#### 四种方式对比

```
同步通信：
  1. REST（HTTP）：最常见，轻量，但吞吐低
  2. gRPC：高性能，支持双向流，用 Protocol Buffers 序列化，比 JSON 快 5-10 倍

异步通信：
  3. MQ（消息队列）：解耦、削峰、可靠
  4. 异步 HTTP（WebSocket / SSE）：服务器推送，实时场景
```

#### REST vs gRPC

```
REST：
  - 用 HTTP + JSON
  - 人类可读，调试方便
  - 浏览器原生支持
  - 单向请求响应

gRPC：
  - 用 HTTP/2 + Protocol Buffers
  - 二进制序列化，体积小速度快
  - 支持双向流（客户端和服务端同时发消息）
  - 需要 proto 文件定义接口
  - 适合内部微服务调用，不适合对外
```

#### MQ 的两个核心场景

```
削峰填谷：
  流量高峰时，MQ 作为缓冲区，Consumer 按自己速度消费
  → 生产者不需要等待消费者处理完

解耦：
  服务 A 发消息到 MQ，服务 B 订阅，不用 A 直接调 B
  → B 挂了，A 照常发，不受影响
```

#### 一句话回答

> "微服务通信分同步和异步：同步用 REST（通用）或 gRPC（高性能内部调用）；异步用 MQ（削峰解耦）或 WebSocket（实时推送）。"

### 7. Redis 持久化 RDB vs AOF

**你犯的错误**：一面回答 Kafka 可靠性时把 Redis 的 AOF 混进去了。

#### RDB（快照方式）

```
原理：定时把内存数据全量dump到磁盘（dump.rdb文件）

优点：
  - 全量快照，恢复快（直接加载文件）
  - 文件紧凑（适合备份）
  - 主进程 fork 子进程处理，不阻塞主进程

缺点：
  - 定时快照，两次快照之间可能丢数据
  - 如果数据量很大，fork 进程会很慢

触发方式：
  - 配置 save 900 1（900秒内1次写就触发）
  - bgsave（后台异步执行）
  - redis-cli save（同步，阻塞）
```

#### AOF（追加日志方式）

```
原理：每次写操作追加到 .aof 文件，恢复时重放所有命令

优点：
  - 每条命令追加，丢数据最多丢 1 秒
  - 写是顺序磁盘 IO，比 RDB 随机写快

缺点：
  - 文件比 RDB 大（存的是命令，不是数据）
  - 恢复慢（需要重放所有命令）

配置：
  appendonly yes
  appendfsync everysec（每秒刷盘，折中方案）
  appendfsync always（每条命令都刷，最安全但最慢）
  appendfsync no（交给 OS，不推荐）
```

#### 两者怎么选

```
数据重要不能丢 → AOF（everysec 或 always）
追求性能，数据丢了可接受 → RDB
两者都开 → AOF + RDB 混合持久化（Redis 4.0+ 支持）
  → 平时用 RDB 快速恢复，间隔用 AOF 补数据
```

#### 一句话回答

> "RDB 是定时快照，恢复快但可能丢数据；AOF 是追加命令日志，最少丢 1 秒数据但文件大、恢复慢。生产环境一般 AOF + RDB 混合用。"

### 8. 布隆过滤器原理

**你暴露的问题**：一面提到了布隆过滤器，但原理说不清。

#### 原理

```
核心思想：用多个哈希函数，把数据映射到一个位数组

插入：数据 x 经过 k 个哈希函数，得到 k 个数组位置，全部设为 1
查询：数据 x 经过 k 个哈希函数，检查这 k 个位置是否全为 1
  - 全为 1 → 可能在集合里（可能误判）
  - 有 0 → 一定不在集合里（绝对准确）
```

#### 误判率推导

```
位数组长度 m，元素个数 n，哈希函数个数 k

误判率 p ≈ (1 - e^(-kn/m))^k

最优哈希函数个数：k = (m/n) × ln2

特点：
  - 元素越多，误判率越高
  - 位数组越大，误判率越低
  - 误判不是漏掉（不在就是不在），而是把不在的判成了在
```

#### 应用场景

| 场景 | 作用 |
|------|------|
| 缓存穿透 | 先查布隆过滤器，不存在直接返回，避免打到 DB |
| 推荐去重 | 用户看过的文章标记，不重复推荐 |
| URL 去重 | 爬虫过滤已爬过的 URL |
| 注册时用户名检查 | 快速判断用户名是否已被占用 |

#### 一句话回答

> "布隆过滤器用多个哈希函数映射到位数组，查询时如果所有位置都是 1 说明可能存在（有误判），有 0 则一定不存在。它牺牲确定性换取极低的空间和查询成本，适合缓存穿透和去重场景。"

### 9. MySQL 分库分表

```
垂直拆分：按业务列分（把大表的字段拆到不同表）
  - 把不常用的列拆出去
  - 把经常一起查询的列放一起

水平拆分：按数据行分（把大表的数据拆到不同库/表）
  - 按 ID 取模：user_id % 4 路由到不同表
  - 按时间：历史数据归档到冷库
  - 按地区：按用户所属区域分库

跨库查询问题：
  - 解决方案：ES（搜索引擎）做全局检索，MySQL 只存明细
  - 或者：封装一层 DBProxy / ShardingSphere 做透明路由
```

### 10. RPC vs MQ 对比

```
核心区别：同步 vs 异步

RPC（同步）：
  - 调用方发请求，等回应才继续
  - 实时性高，但强耦合
  - 超时/被调用方挂了会直接影响调用方

MQ（异步）：
  - 发完就不管了，等消费者来取
  - 解耦，被调用方挂了不影响调用方
  - 但不能获取实时返回值

选择原则：
  - 需要立刻拿到结果 → RPC
  - 不需要立刻拿到结果，允许延迟 → MQ
  - 削峰填谷 → MQ
  - 服务状态强一致性（如下单）→ RPC + 消息事务
```

## FunASR & FunAudioChat 论文核心（跟你的项目直接相关）

> 这两篇是你项目里实际调用的模型的技术报告，二面可能问到模型选型和原理。

---

### Fun-ASR 核心要点

#### 模型架构

```
Fun-ASR = Audio Encoder(0.7B) + Audio Adaptor(2层Transformer) + LLM(7B Qwen)
Fun-ASR-nano = Audio Encoder(0.2B) + LLM(0.6B)

额外：CTC decoder 用于 hotword 定制检索
```

#### 训练流程（5阶段 SFT）

```
Stage 1：冻住 encoder + LLM，只训练 Adaptor（对齐音频→LLM语义空间）
         数据：200k 小时，70K 步

Stage 2：冻住 LLM，训练 encoder + Adaptor（学更好的语义表示）
         数据：10M 小时，1 epoch

Stage 3：冻住 encoder + Adaptor，用 LoRA 微调 LLM（防灾难性遗忘）
         数据：20K 小时

Stage 4：全参数微调 encoder + Adaptor + LLM(LoRA)
         数据：3M 小时（高质量，三个 ASR 模型共同评估过）

Stage 5：冻住 encoder，只训练 CTC decoder（用于 hotword 检索增强）
```

#### RL 强化学习（FunRL 框架）

```
核心算法：GRPO（Group Relative Policy Optimization）

奖励函数设计（对面试有用）：
  R1：ASR 准确率 = 1 - WER
  R2：关键词召回 + 关键词准确率（trade-off）
  R3：幻觉抑制（噪声环境下模型容易生成不存在的内容，打击幻觉）
  R4：语言一致性（输出语言必须和输入语言一致，否则 -1）

RL 训练数据 100K 样本，分五类：
  - Hardcase：三个外部 ASR 系统一致但 Fun-ASR 不一致的样本
  - 长语音：>20 秒的片段
  - 幻觉样本：模型容易产生幻觉的数据
  - 关键词样本
  - 普通 ASR 数据（防止灾难性遗忘）
```

#### 生产优化

```
1. 流式 ASR：构造流式风格的训练数据，减少训练-推理不匹配
2. 噪声鲁棒：SNR=10dB 合成噪声数据，相对提升 ~13%
3. 混语（中英切换）：用 Qwen3 生成中英混合文本 + TTS 合成
4. Hotword 定制：CTC hypothesis → phoneme 匹配 → RAG 增强
5. 幻觉抑制：训练时注入纯噪声片段（zero-padding），让模型学会"没声音就不输出"
```

#### 一句话回答面试官

> "Fun-ASR 是一个 LLM-based ASR 系统，Audio Encoder 提取音频特征，Adaptor 对齐到 LLM 语义空间，Qwen 作为 LLM decoder 生成文本。它的训练分 5 阶段 SFT，最后用 GRPO 做强化学习来抑制幻觉和提升关键词召回。"

### FunAudioChat 核心要点

#### 核心问题：语音 token 和文本 token 的时间分辨率不匹配

```
语音 token：25Hz（每秒 25 个 token）
文本 token：约 3Hz（每秒 3 个 token）

问题：25Hz 直接进 LLM，计算量巨大，语义被稀释

FunAudioChat 解决方案：DRSR（双分辨率语音表示）
  - LLM 处理：降采样到 5Hz（5 个 token/秒）→ 计算量降 ~50%
  - 生成：Speech Refined Head 恢复 25Hz 高质量语音 token
```

#### 训练策略：Core-Cocktail Training（防灾难性遗忘）

```
核心矛盾：
  学习率太高 → 破坏 LLM 原有知识（灾难性遗忘）
  学习率太低 → 收敛太慢

两阶段解法：
  Stage 1：高学习率（1e-4→1e-5）快速适配多模态
  中间：模型合并 M_r = 0.5*M_Stage1 + 0.5*M_Pretrained
  Stage 2：低学习率（1e-5→1e-6）精细调优

效果：既能快速适应音频任务，又能保留原始文本 LLM 能力
```

#### 关键训练阶段

```
Pre-alignment：冻住 LLM，对齐 Audio Encoder + Adaptor + SRH
Core-Cocktail：解决灾难性遗忘（上面讲的两阶段）
Multi-Task DPO：偏好学习增强鲁棒性、指令遵循、情感共情
Full-Duplex：支持双向同时对话（打断、接话等）
```

#### 一句话回答面试官

> "FunAudioChat 是一个大语音语言模型（LALM），解决的核心问题是语音 token（25Hz）和文本 token（3Hz）的分辨率不匹配。它用双分辨率架构：LLM 以 5Hz 处理语音降计算量，Speech Refined Head 生成 25Hz 高质量语音。训练用 Core-Cocktail 两阶段防止灾难性遗忘。"
