## 二、FunAudioLLM / Fun-Audio-Chat 核心优势

> 本轮新增整理的内容，参考官方 GitHub + arxiv Technical Report (2025.12)。

### 架构创新：双分辨率语音表示（DRSR）

```
输入音频 → Codec Encoder → 共享 Backbone (5Hz) → LLM 理解
                                            ↓
                               Speech Refined Head (SRH) → 25Hz 语音生成

GPU 计算量：比纯 25Hz 模型减少约 50%
```

- 输入端低帧率（5Hz）减少 token 数加速理解
- 生成端用 SRH 恢复到 25Hz 保证音质
- 一进一出不同分辨率，效率与质量兼得

### 全双工对话（Full-Duplex）

- Fun-Audio-Chat-Duplex 支持同时听、同时说
- 自然打断和接话，不需要等对方说完
- 端到端建模，不依赖人工设计的轮次控制信号

### 训练策略：Core-Cocktail Training + Multi-Task DPO

- Stage 1: SFT（保留文本 LLM 能力）+ Merge（多专家权重合并）
- Stage 2: Multi-Task DPO（鲁棒性/指令跟随/语音理解/声音情感四维对齐）

### 和同类模型横向对比

| 模型 | 输入帧率 | 输出帧率 | 核心优势 |
|------|---------|---------|---------|
| **Fun-Audio-Chat-8B** | **5 Hz** | **5 Hz** | GPU 算力节省 50%，SOTA 口语 QA |
| GLM-4-Voice (9B) | 12.5 Hz | 12.5+τ | 标准 LALM 架构 |
| MiniCPM-o 2.6 (7B) | 25 Hz | τ | 高频处理 |
| Kimi-Audio (7B) | 12.5 Hz | 12.5 | 开源竞争基准 |
| MiniMo-Audio (7B) | 6.25 Hz | 6.25+τ | 低频处理 |

**关键结论**：以最低输入帧率（5Hz）实现同档次最强性能，训练和推理成本最低。

### 在项目里的定位

## 三、本轮知识问答总结（给下一个 AI 看的核心内容）

### 1. MHA 头数设计

**结论：不是越多越好，需要在「表达能力」和「计算效率」之间权衡。**

```
head 越多 → d_k = d_model / num_heads 越小
           → d_k < 64 时，单个 head 表达能力急剧退化
```

| 方案 | Head 数 | d_k | KV 头数 | 显存 | 表达能力 |
|------|---------|-----|---------|------|---------|
| MHA | = num_heads | d/n | = num_heads | O(n×L×d) | 强，显存最大 |
| MQA | = num_heads | d/n | **= 1** | O(1×L×d) | 弱，多头退化 |
| GQA | = num_heads | d/n | **= n/g** | O(n/g×L×d) | 平衡，推荐 |

> "Qwen2.5 的设计原则是保证 d_k >= 64。比如 72B 模型 d_model=8192，num_heads=64，d_k=128，分到 8 个 KV head 后每个 KV head 维度仍是 128，足够表达语义。"

### 2. OpenCLIP 流水线结构

- **双塔架构**：Image Encoder（ViT/ConvNeXt/ResNet）+ Text Encoder（Transformer/GPT2-style）
- **训练目标**：Contrastive Learning，N×N cosine similarity 矩阵，对角线为正样本
- **Loss**：InfoNCE / NT-Xent，同时优化 image→text 和 text→image 两个方向
- **最新版本 v3.3.0 (2026.02)**，支持 SigLIP2、CLIPA 等新模型

### 3. YOLO 从 v11 开始的发展史

```
YOLO11 (2024.09) → YOLOv12 (2025.02) → YOLOv13 (2025.06) → YOLO26 (2025.09)
核心演进：
  YOLO11: C3k2 + C2PSA（效率优化）
  YOLOv12: R-ELAN + Area Attention（注意力机制引入）
  YOLOv13: HyperACE + FullPAD（超图全局建模）
  YOLO26: DFL移除 + NMS-Free + ProgLoss + STAL + MuSGD（端侧最优，CPU 快 43%）
```

**为什么选 YOLO11**：
- C3k2 比 C2f 减少 30% 参数，小物体 mAP ↑2~3%
- C2PSA 并行空间注意力，低代价增强特征聚焦
- Ultralytics 官方生态完整，训练/部署/导出无缝衔接
- YOLO11n 在 CPU 上 56ms，比 v8n 快 30% 且精度更高

### 4. PageAttention V2 相对于 V1 的改进

**本质：V1 基于 FlashAttention-2，V2 基于 FlashAttention-3 + Paged KV Cache**

| 指标 | V1 | V2 |
|------|-----|-----|
| KV Cache | 连续分配，预分配 | 分页管理，动态分配 |
| 显存利用率 | ~60-70% | ~90%+ |
| Decode 吞吐 | 基准 | **2-4x 提升** |
| 长序列 Prefill | 正常 | **1.5-2x 提升** |
| 硬件 | Ampere+ | **Hopper+ (H100)** |
| 量化 | FP16/BF16 | **FP8 + FP16/BF16** |

> "V2 让显存管理从'预分配连续块'变成'按需分页'——V1 像一次性订 100 间连续酒店房，中间退房无法复用；V2 像 Airbnb 按需订单间，退房后立即可被新客使用。"

## 六、RAG 两阶段检索

```
Stage 1: Bi-Encoder 粗排
  query → BGE Embedding → ChromaDB → Top-15

Stage 2: Cross-Encoder 精排
  (query, doc) 拼接 → BGE Reranker → Top-3

Stage 3: Soft Decay 时间衰减
  final_score = rerank_score × √(decay)
  半衰期 7 天，关键词豁免
```

## 六补充知识：BGE 两个模型的基座区别（BERT vs XLM-RoBERTa）

### BERT → RoBERTa → XLM-RoBERTa 三代关系

```
BERT-Large:   340M  / 24层 / 1024维 / 16头 / 英文 16GB / 学习型 PE
    ↓ 10倍数据 + 去掉NSP + 动态Mask + 更长训练
RoBERTa-Large: 355M  / 24层 / 1024维 / 16头 / 英文 161GB / 学习型 PE
    ↓ 100语言 + 2.5TB多语言语料
XLM-RoBERTa-Large: 560M  / 24层 / 1024维 / 16头 / 100语言 2.5TB / 学习型 PE
```

**RoBERTa vs BERT 的核心改进：**
- 去掉 NSP（Next Sentence Prediction）任务，对下游任务无稳定提升
- 数据量扩 10 倍（16GB → 161GB）
- 动态 Mask（每个 epoch 重新生成，而非一次性静态 Mask）
- 更大 batch（8K vs 256）+ 更长训练步数
- 结果：355M 参数在 GLUE/SQuAD 全面超越 BERT

**XLM-RoBERTa vs RoBERTa 的核心改进：**
- 100 种语言联合 MLM 预训练，词表扩到 250k
- 总预训练数据 2.5TB（CommonCrawl）

**为什么两个 BGE 模型用不同基座：**
- BGE-Embedding-Large → BERT-style 中文词表（30k）：中文专用，效率高
- BGE-Reranker-Large → XLM-RoBERTa（560M）：Cross-Encoder 需要更强跨语言泛化和全文交互建模

> "BGE-Reranker 用 XLM-RoBERTa 不是因为中文不够好，而是 Cross-Encoder 做精排时需要更全面的语义交互能力，XLM-RoBERTa 的多语言预训练让它在中文 Cross-Encoder 任务上比纯中文 BERT 效果更好。"
