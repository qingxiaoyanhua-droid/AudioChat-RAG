# 面试准备文档 - DiTX-Clerk 智能会议系统

## 1 分钟项目介绍

> "我开发了一个多模态语音对话系统 AudioChat-RAG。核心流程是：输入音频经过说话人分离、ASR 识别后，通过 RAG 检索历史会议记录增强 LLM 生成，最终输出文本或语音回复。
>
> 技术亮点：
> 1. **RAG 检索增强** - ChromaDB+BGE 语义检索，支持时间衰减重排序
> 2. **GRPO 强化学习** - 多维度评估器（CER+ 流畅度 + 相关性）
> 3. **端到端优化** - 从音频到回复的全流程自动化
>
> 实验结果显示，RAG 将回答准确性提升了约 31%。"

## 核心技术详解

### 1. RAG 检索增强（面试重点⭐⭐⭐⭐⭐）

**技术方案:**
- 向量数据库：ChromaDB（轻量级，支持持久化）
- 嵌入模型：BGE-large-zh-v1.5（中文优化）
- 检索策略：语义检索 + 说话人过滤 + 时间衰减重排序

**时间衰减公式:**
```python
decay = 0.5 ** (days_diff / half_life_days)  # half_life = 7 天
final_score = relevance_score * decay
```

**效果提升:**
| 指标 | 无 RAG | 有 RAG | 提升 |
|------|--------|--------|------|
| 回答准确性 | 0.64 | 0.84 | **+31%** |
| 信息完整性 | 0.58 | 0.79 | **+36%** |

**面试回答示例:**
> Q: RAG 检索中时间衰减是怎么设计的？
>
> A: 我采用了指数衰减函数，半衰期设为 7 天。最近的会议记录权重更高，因为项目进度信息时效性强。公式是 `decay = 0.5^(days/7)`，最小值为 0.1 避免完全忽略旧数据。

### 1.5 分层 RAG 设计（面试重点⭐⭐⭐⭐）

在传统 RAG 基础上，引入了两层知识聚合：

**L2 人物画像 — 从会议记录中聚合团队成员画像**

```
L2PersonProfile {
    name, role, department
    permissions: ["commit", "approve_pr"]      # 权限信息
    active_tasks: ["#123", "#456"]            # 当前任务
    dominat_topics: ["API联调", "需求评审"]     # 高频讨论话题
    speaking_ratio: 0.35                      # 发言占比
    frequent_collaborators: ["张三", "李四"]    # 协作关系
}
```

生成逻辑：从 FunASR 说话人分离结果中统计发言时长、话题分布、协作关系，自动聚合生成。

典型 Query："后端这块找谁？" → 按 role=后端 + active_tasks 最多的 profile 返回。

**L3 需求演进轨迹 — 追踪需求从提出到上线的全生命周期**

```
L3RequirementEvolution {
    requirement_id: "REQ-2024-001"
    title: "支付模块重构"
    current_phase: "testing"
    milestone_timestamps: {
        proposed: "2024-01-10",
        approved: "2024-01-15",
        dev_started: "2024-01-20",
        dev_done: "2024-02-28",
        testing: "2024-03-05"
    }
    version_history: [
        {"version": "v1", "summary": "方案A", "meeting_id": "..."},
        {"version": "v2", "summary": "方案B（加了缓存）", "meeting_id": "..."}
    ]
    blockers: [
        {"type": "排期冲突", "description": "前端排期冲突", "resolved": True}
    ]
}
```

生成逻辑：每场需求评审会的 action_item 自动关联到 requirement_id，跨会议聚合版本变更和阻塞记录。

典型 Query："这个需求现在到哪了？" → 返回 current_phase + 最近里程碑时间戳。

**存储分离：**
- L2 事实条目 + L3 SOP → ChromaDB（向量检索）
- L2 人物画像 → JSON 文件（独立实体存储）
- L3 需求演进轨迹 → JSON 文件（独立实体存储）

**为什么不用 ChromaDB 存人物画像和需求轨迹？**
- ChromaDB 主语是"chunk"，适合检索；人物画像主语是"人"，需求演进主语是"需求"
- 两者需要精确查询（按 name、按 requirement_id）而非向量相似度检索
- 独立 JSON 存储，查询更简单，结构更灵活

**面试回答示例:**
> Q: 你的 RAG 有几层？L2/L3 存的是什么？
>
> A: 我设计了三层知识体系。L1 是索引层始终注入。L2 有两条线：ChromaDB 存事实条目（项目背景、决策、action_items），JSON 文件存人物画像（从会议中聚合的团队成员角色、任务、协作特征）。L3 也是两条线：ChromaDB 存 SOP 流程知识，JSON 文件存需求演进轨迹（追踪每个需求从提出到上线经历了哪些变更、卡在哪里）。这样分工明确，检索层用向量搜索，实体层用结构化查询。

### 2. GRPO 强化学习（面试重点⭐⭐⭐⭐⭐）

**GRPO vs PPO:**
| 特性 | PPO | GRPO |
|------|-----|------|
| Critic 模型 | 需要 | 不需要 |
| 优势计算 | Value 网络估计 | 组内相对优势 |
| 实现复杂度 | 高 | 低 |
| 计算效率 | 较低 | 较高 |

**GRPO 核心公式:**
```python
# 组内相对优势（GRPO 核心创新）
A_i = (r_i - mean(r)) / std(r)

# GRPO 损失
L = -E[log(π_new/π_old) * A_i] + kl_coef * KL(π_new || π_ref)
```

**多维度奖励函数（7维度）：**
```python
reward = (0.30 * accuracy
         + 0.25 * faithfulness
         + 0.15 * fluency
         + 0.10 * relevance
         + 0.15 * structure
         + 0.05 * length
         + 0.05 * dependency_dag)

# 准确性（30%）：与 reference 的 BGE 余弦相似度
# 忠实度（25%）：逐句与 RAG 原文比对，防止幻觉
# 流畅度（15%）：n-gram 重复度 + 长度惩罚
# 相关性（10%）：与 query 的语义相似度
# 结构规范（15%）：Markdown 结构完整 + action_items owner 非空
# 长度合规（5%）：在合理字数范围内
# DAG 环检测（5%）：Kahn 算法检测 action_items 依赖图中是否存在环
#   - 有环 → 0.0（任务互相等待死锁）
#   - 无环 → 1.0
```

**训练配置:**
- 模型：Fun-Audio-Chat-8B / Qwen-7B
- GPU: 8×A100 80GB
- 数据量：1000+ 会议纪要偏好对
- 学习率：1e-6 (GRPO 阶段)
- Group size: 4（每个 prompt 生成 4 个 response）

**面试回答示例:**
> Q: GRPO 的奖励函数如何设计？
>
> A: 我设计了 7 维度加权奖励：准确性 30%（与参考文本的嵌入相似度）、忠实度 25%（逐句和 RAG 原文比对防止幻觉）、流畅度 15%（n-gram 重复度）、相关性 10%（与 query 的语义相似度）、结构规范 15%（Markdown 完整 + owner 非空）、长度合规 5%、DAG 环检测 5%（Kahn 算法判断依赖图是否有环）。内容质量和事实准确性占 55%，这是会议纪要生成任务的核心。

### 3. 端到端 Pipeline（面试重点⭐⭐⭐⭐）

**模块组成:**
1. **说话人分离** - 3D-Speaker (CAM++ 声纹嵌入 + VAD 语音检测)
2. **ASR 识别** - FunASR (Paraformer-large, 16kHz)
3. **RAG 检索** - ChromaDB + BGE 语义检索
4. **LLM 生成** - Fun-Audio-Chat-8B
5. **TTS 合成** - CosyVoice-0.5B（可选）

**流式处理:**
- 音频切片：500ms 窗口
- 边说话边转录
- 延迟：<2s（首字）

**面试回答示例:**
> Q: 如何处理长音频？
>
> A: 采用流式处理架构，500ms 切片并行处理。VAD 检测语音段后，每个片段独立进行说话人分离和 ASR 识别，最后按时间戳拼接。这样突破了单次处理长度限制，同时保持低延迟。

## 面试常见问题与回答

### RAG 相关

**Q1: 为什么选择 ChromaDB？**
> A: ChromaDB 轻量级、易部署，支持持久化存储和高效的近似最近邻搜索。对于会议记录这种中等规模数据（万级文档），ChromaDB 性能足够且运维成本低。相比 Milvus、Weaviate 等重型方案，更适合内部工具快速迭代。

**Q2: 如何处理说话人分离的时序信息？**
> A: 每个 utterance 保存 start_ms 和 end_ms 毫秒级时间戳，检索时作为元数据存储。时间衰减重排序时，根据 timestamp 计算 decay 因子，确保近期会议权重更高。

### GRPO 相关

**Q3: GRPO 和 DPO 的区别？**
> A:
> - DPO: 直接优化偏好数据 (chosen/rejected)，不需要奖励模型
> - GRPO: 基于组内相对优势优化，需要设计奖励函数
> - GRPO 优势：更灵活，可以动态调整奖励维度

**Q4: 训练不稳定怎么办？**
> A:
> 1. 调小学习率（1e-6 → 5e-7）
> 2. 增加 KL 惩罚（0.1 → 0.2）
> 3. Gradient clipping（max_norm=1.0）
> 4. Warmup + cosine decay 调度

### Pipeline 相关

**Q5: ASR 转写准确率如何达到 92%？**
> A:
> 1. 使用 Paraformer-large 大模型
> 2. 16kHz 高质量音频输入
> 3. VAD 精准切分语音段
> 4. 后处理标点恢复模型

**Q6: 如何处理多人同时说话？**
> A: 3D-Speaker 的 CAM++ 模型提取声纹嵌入，通过聚类算法区分说话人。重叠语音通过 VAD 检测和能量分析处理，优先保留能量较高的语音段。

## 手撕代码准备

### GRPO 优势计算（必考）

```python
def compute_advantage(self, rewards: List[float]) -> torch.Tensor:
    """组内相对优势计算"""
    rewards = torch.tensor(rewards, dtype=torch.float32)
    mean_reward = rewards.mean()
    std_reward = rewards.std() + 1e-8
    advantages = (rewards - mean_reward) / std_reward
    return advantages
```

### 时间衰减函数

```python
def time_decay(self, timestamp: str, half_life_days: float = 7.0) -> float:
    """指数时间衰减"""
    doc_time = datetime.fromisoformat(timestamp)
    now = datetime.now()
    days_diff = (now - doc_time).days
    decay = 0.5 ** (days_diff / half_life_days)
    return max(0.1, decay)
```

### 流畅度评估

```python
def fluency_reward(self, text: str) -> float:
    """基于 n-gram 重复度"""
    ngrams = [text[i:i+3] for i in range(len(text)-2)]
    if not ngrams:
        return 0.0
    unique_ratio = len(set(ngrams)) / len(ngrams)

    # 长度惩罚
    length = len(text)
    if length < 10 or length > 500:
        length_penalty = 0.5
    else:
        length_penalty = 1.0

    return unique_ratio * length_penalty
```
