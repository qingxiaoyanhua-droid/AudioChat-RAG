# DiTX-Clerk 项目全貌

> 一句话了解项目是什么、怎么跑的、有哪些模块、核心指标是什么。

---

## 一句话总结

**面向企业会议场景的智能转录与分析系统**：音频 → 说话人分离 → ASR → 意图识别路由 + RAG 检索增强 → LLM 生成结构化会议纪要 → 任务派发（Gitea API）

---

## 完整技术链路

```
会议音频
  ↓
16kHz Mono 预处理
  ↓
说话人分离（3D-Speaker, CAM++ + VAD）
  ↓
ASR 转写（FunASR Paraformer，准确率 92%）
  ↓
意图识别路由（Phase 2，新增）
  ├─ 问"这场会议"→ meeting_id_filter 精确检索
  └─ 问"上次"/"上周"→ time_range 过滤
  ↓
RAG 检索（Phase 1-3）
  ├─ Bi-Encoder 粗排（ChromaDB Top-15）
  ├─ Cross-Encoder 精排（BGE Reranker）
  ├─ Soft Decay（decay = 0.5^(days/7)）
  └─ Agentic RAG（Phase 3：信息缺口分析 → 并发检索 → 上下文压缩）
  ↓
LLM 生成（Fun-Audio-Chat-8B）
  ↓
HITL 质量报告（AI 自评 + 人工审核）
  ↓
输出：结构化会议纪要 + 行动项 + 决策项 + Gitea 任务派发
```

## 两个模型的职责区分

| 模型 | 职责 | 是否微调 | 面试重点 |
|------|------|---------|---------|
| Fun-Audio-Chat-8B | ASR 语音转文字 + 说话人分离 | **否**，冻结 | 基础组件，不展开 |
| Qwen2.5-0.5B-Instruct | SFT + GRPO 微调，生成会议纪要 | **是** | 面试核心 |

## 三大技术模块

| 模块 | 实现方式 | 核心文件 |
|------|---------|---------|
| **RAG 检索增强** | ChromaDB + BGE 两阶段 + Soft Decay + 意图路由 | `audiochat/rag/retriever.py` |
| **GRPO 强化学习** | 自研 GRPOTrainer（无 Critic）+ 9维奖励 + 软 gate | `train_grpo.py`, `grpo_reward_function.py` |
| **SFT 监督微调** | Transformers Trainer + LoRA | `train_sft.py` |

## 项目亮点（面试 1 分钟介绍用）

> "我开发了一个多模态语音对话系统 DiTX-Clerk，面向企业会议场景。
>
> 核心流程：输入音频经过说话人分离、ASR 识别后，通过两层 RAG 检索增强 LLM 生成——第一层是意图识别路由，根据用户问题判断是查当前会议还是历史会议；第二层是时间感知的语义检索，用 Soft Decay 处理信息新鲜度。最后 LLM 生成结构化的会议纪要，包含行动项和决策项。
>
> 技术亮点：
> 1. **意图识别路由** — 规则兜底 + 小模型精判，解决'当前会议 vs 历史会议'的分流问题
> 2. **Agentic RAG** — LLM 主动分析信息缺口，并发批量检索补充历史背景
> 3. **9维 GRPO 奖励** — 准确性、忠实度、流畅度、依赖拓扑验证、可执行性等
>
> 实验结果显示，RAG 将回答准确性提升了 31%。"

## 关键面试话术

### 为什么用 GRPO 不用 DPO？
> "会议纪要的评价维度是明确的——准确性、结构化、完整性，适合用规则量化。DPO 需要大量 chosen/rejected 偏好对，GRPO 在线采样+奖励函数更高效可控。"

### 时间衰减半衰期 7 天怎么定的？
> "企业周会一周一次，一周前内容相关性降一半符合业务直觉。我跑了 3/7/14 天消融：3 天衰减太快丢失历史信息，14 天近期优先不足，7 天最优（Top-3 准确率 85%）。"

### RAG 什么时候会失效？
> "1. 知识库为空——降级到纯 LLM 生成；2. 检索质量差——扩展到关键词混合检索；3. 知识冲突——时间优先取最新。"
