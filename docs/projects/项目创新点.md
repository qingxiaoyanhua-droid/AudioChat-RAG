# DiTX-Clerk 项目创新点与技术贡献

> 面试时可背诵的 3–5 条创新点，用于回答「你这个项目有啥创新？」

---

## 一、核心创新点（必背）

### 1. 会议场景的时间感知检索

**问题**：传统 RAG 只做语义相似度排序，不区分新旧信息。会议场景下，用户问「最近进度」时应优先近期记录，问「架构决策」时应保留历史重要信息。

**方案**：
- **时间衰减**：`decay = 0.5^(days/7)`，half_life=7 天
- **关键词豁免**：对「架构、决策、负责人」等 timeless 词，不施加时间衰减
- **消融验证**：去掉时间衰减后，对「最近进度」类 query 准确率跌约 12%

**面试话术**：「RAG 本质是检索+排序，我们加了时间维度，相当于搜索里的 freshness 权重。」

---

### 2. 多任务 GRPO 奖励函数

**问题**：单一 reward 容易导致奖励黑客（如堆关键词、拉长度）。

**方案**：
- 拆成 **纪要质量 + 行动项质量 + 决策项质量 + 整体格式** 四个子任务
- 权重：0.35 + 0.35 + 0.20 + 0.10
- 各子任务用 BGE 语义相似度、召回率、格式检查等分别计算

**面试话术**：「多任务 reward 避免模型只优化一个维度，生成更均衡。」

---

### 3. 端到端流式架构

**问题**：长音频一次性处理延迟高，用户需长时间等待。

**方案**：
- **500ms 切片**：边说话边转录，首 token 延迟大幅降低
- **流式 ASR → 增量 LLM**：不攒完整文本再生成

**面试话术**：「会议处理时长缩短 30%，可扩展到实时场景。」

## 二、技术贡献总结

| 贡献 | 类型 | 与 TikTok 搜索的关联 |
|------|------|----------------------|
| 时间感知检索 | 算法设计 | 语义召回 + 时间衰减 ≈ 搜索 freshness |
| 多任务 GRPO | 训练方法 | 多目标优化思路可迁移 |
| 流式架构 | 系统设计 | 低延迟、可扩展 |
| RAG 检索增强 | 工程实践 | 检索排序是搜索核心 |

## 三、差异化叙事（区别于「做过 SFT/RL」）

| 同质化说法 | 本项目说法 |
|------------|------------|
| 「用了 LoRA 微调」 | 「0.5B 全量易过拟合，LoRA 够用；做了格式统一和长度过滤」 |
| 「用了 GRPO」 | 「多任务 reward 拆成纪要/行动项/决策项，避免奖励黑客」 |
| 「接了 RAG」 | 「时间衰减 + 关键词豁免，RAG 本质是检索排序问题」 |

## 四、可现场演示的硬核问题

1. **时间衰减半衰期 7 天怎么定的？**  
   → 业务假设（周会频率）+ 简单消融，可继续调参。

2. **奖励函数权重怎么来的？**  
   → 人工调参 + 验证集上观察与人工打分相关性。

3. **31% 提升怎么算的？**  
   → 50 条标注 query，BGE 语义相似度（生成 vs 标准答案），可复现。

4. **Top-3 检索 83% 怎么算的？**  
   → 至少 1 个相关即算对，50 条 query 统计。
