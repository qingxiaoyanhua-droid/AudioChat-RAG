## 一、项目是什么

**DiTX-Clerk** 是一个面向企业会议场景的智能转录与分析系统。

```
会议音频
  ↓
说话人分离（3D-Speaker, CAM++）
  ↓
ASR 转写（FunASR Paraformer，准确率 92%）
  ↓
意图识别路由（Phase 2，新增）
  ↓
RAG 检索增强（Phase 1-3，持续演进）
  ↓
LLM 生成（Fun-Audio-Chat-8B）
  ↓
HITL 质量报告（AI 自评 + 人工决策）
  ↓
输出：结构化会议纪要 + 行动项 + 决策项 + 任务派发（Gitea API）
```

## 二、今天（2026-06-03）做了什么

本次会话补全了 RAG 系统的两个缺失环节：**意图识别** 和 **Agentic RAG**。

### Phase 1：Metadata Filter 修复

**问题**：`meeting_id` 和 `timestamp` 写入了 ChromaDB，但检索时没有传，候选集偏大。

**改动**：
- `storage.py`：`search()` 支持 `meeting_id_filter` + `time_range` 复合 ChromaDB where 过滤
- `retriever.py`：`retrieve()` 透传新参数，缓存 key 加上 filter 维度

### Phase 2：意图识别路由

**问题**：QA 模式下用户问题直接拿去检索，不知道用户是想查"当前会议"还是"历史会议"。

**方案**：规则兜底 + 小模型精判，3 类意图分流。

| 意图 | 触发 | 检索策略 |
|------|------|---------|
| `CURRENT_MEETING` | 含"这场会议""刚才" | `meeting_id_filter=当前会议ID` |
| `CROSS_MEETING` | 含"上次""上周""之前" | `time_range=(start, end)` |
| `UNKNOWN` | 规则未命中 | 默认全量 RAG |

- 规则 ~0ms，命中直接返回
- 未命中走 Qwen2.5-0.5B-Instruct（已加载）+ `@lru_cache` 避免重复推理
- 实现位置：`audiochat/prompting.py` 的 `classify_intent()` + `route_and_retrieve()`
- QA 模式接入：`build_llm_instruction()` 内部调用意图路由

### Phase 3：Agentic RAG

**问题**：总结模式只取最后一句做检索，没有分析转写内容、主动补充历史信息。

**方案**：LLM 分析信息缺口 → 并发批量检索 → 上下文压缩 → 生成总结。

```
转写文本
  ↓
analyze_info_gaps()    [LLM 调用 1]
  ↓ InfoGap 列表（info_type, topic, sub_query）
plan_and_retrieve()    [并发，max_workers=4]
  ↓ 检索结果去重
compress_contexts()    [按层级/score 排序，截断至 ~2000 tokens]
  ↓
build_agentic_instruction() [拼接]
  ↓
Fun-Audio-Chat-8B 生成  [LLM 调用 2]
```

关键实现：
- 并发：用 `ThreadPoolExecutor` 并发执行所有子查询，总耗时 ≈ max(各查询) 而非 sum
- 压缩：L3 SOP > L2 事实 > FALLBACK > L1 > RAW，按 relevance_score 截断
- 触发：`--agentic-rag` 参数，在 `stage_llm()` 中判断是否启用
- 新增参数：`--time-range 2026-05-01,2026-06-03`

### 附带修复

`stage_llm()` 原来没有 return 语句，但 `main()` 里在用 `llm_result.text`，这是一个静默 bug。已修复。

## 三、技术决策记录（本次会话）

### 1. 意图识别为什么用规则 + 小模型，不用纯规则或纯模型？

| 方案 | 延迟 | 准确率 | 成本 |
|------|------|--------|------|
| 纯规则 | ~0ms | 低，歧义必漏 | 0 |
| 纯小模型 | ~200ms | 高 | 已有 0.5B 模型在用，不额外花钱 |
| **规则兜底 + 小模型精判** | ~0ms（命中时）| 高 | 0 |

选择了第三种：规则处理 90% 的明显场景，模糊场景走小模型。

### 2. Agentic RAG 为什么用并发而不是串行？

串行：分析缺口 1 → 检索 → 分析缺口 2 → 检索 → ... → 总耗时 = sum(各阶段)

并发：一次性分析所有缺口 → 并发执行所有检索 → 总耗时 ≈ max(各查询)

ThreadPoolExecutor 4 个 worker，最多同时跑 4 个检索，总耗时从线性叠加变成取最大值。

### 3. 上下文压缩为什么用启发式而非 LLM？

LLM 压缩需要额外一次 LLM 调用（约 500ms），且输出不稳定。

启发式压缩（按层级 + score 排序 + 字符数截断）零开销、稳定可复现，且效果足够——高相关的 L3/L2 内容会被优先保留。

## 五、面试怎么讲（本次新增部分）

### 意图识别路由

> "RAG 检索前加了一层意图识别。用户问'这场会上张三说了什么'，用规则判断是'当前会议'意图，直接在当前会议 ID 下精确检索；用户问'上周进度'，判断是'历史会议'意图，用时间范围过滤。规则命中占 90% 以上，模糊情况走小模型分类，两个意图分流策略覆盖了绝大部分场景。"

### Agentic RAG

> "总结模式下，LLM 不是直接生成，而是在生成前先分析这场会议里哪些议题需要补充历史信息，输出结构化的信息缺口列表，每个缺口是一个子查询。比如识别出'X模块进度''李四对Y的看法'两个缺口，就同时发两个检索请求，总耗时是取 max 而不是 sum。最后把所有检索结果按层级和相关性排序压缩，塞入上下文再生成。两轮 LLM 调用比一次塞入超长文本更稳定，幻觉也更少。"
