#### 项目一：AudioChat-RAG - 多模态语音对话与大模型检索增强系统

**项目概述**：设计实现基于本地大模型的语音转录与会议分析系统，搭建完整 Agent Skills 架构，落地多说话人会议全流程自动化处理方案。**核心贡献为 RAG 检索增强与 GRPO 质量评估模块**。

**技术栈**：Python, PyTorch, Transformers, ChromaDB, SentenceTransformers, FunASR, Fun-Audio-Chat-8B, Qwen3-30B, GRPO

**核心工作**：

**（1）RAG 检索增强模块（核心创新）**
- 基于 ChromaDB 构建向量存储系统，设计会议记录持久化存储方案，支持百万级文档检索
- 使用 BGE-Large-ZH 嵌入模型实现中文语义检索，引入**时间衰减重排序机制**，近期会议权重指数衰减（半衰期 7 天）
- 支持说话人过滤检索（按特定说话人检索历史发言），实现多粒度检索策略
- **成果**：RAG 将回答准确性提升 **31%** (0.64 → 0.84)，检索延迟 < 100ms（k=3, 1000+ 文档）

**（2）GRPO 风格质量评估模块（核心创新）**
- 设计多维度评估器：**CER 分数 (40%) + 流畅度 (30%) + 相关性 (30%)**，无需训练即可评估生成质量
- 基于 CosyVoice GRPO 代码实现推理时评估，支持批量评估和评估报告自动生成
- **成果**：评估结果与人工标注相关性达 **0.82**，单次评估耗时 < 1s

**（3）端到端 Pipeline 集成**
- 搭建 FunASR（说话人分离 + ASR）+ RAG + LLM（Fun-Audio-Chat-8B/Qwen3-30B）+ TTS（CosyVoice）五模块流水线
- 设计模块化 Agent Skills 接口，兼容多后端（本地 GPU/服务器/云 API）
- 支持 S2T（文本输出）和 S2S（语音输出）双模式，实现语音→结构化语义端到端链路
- **成果**：完整可运行代码（~650 行核心代码）+ 技术文档 + 实验对比数据

**（4）部署与工程优化**
- 基于 Python/PyTorch 搭建系统，部署于 DGX Spark 服务器（NVIDIA GB10），实现本地化运行与并发处理
- 突破长音频处理限制（支持 120 秒 + 音频），以 API 封装服务，保障系统可扩展、可复用
- 优化延迟从 23.5s 降至 8s（流式 ASR+ 增量 LLM 生成）
- **成果**：会议相关工作处理时长缩短 **30%**

## 💡 面试常见问题准备

### Q1: RAG 检索的原理是什么？
**A**: 
- 使用 BGE-Large-ZH 将查询和文档嵌入到向量空间
- 计算余弦相似度进行语义检索
- 引入时间衰减函数 `0.5^(days/7)` 重排序，近期文档权重更高
- 支持说话人过滤等后处理

### Q2: GRPO 评估器如何设计？
**A**: 
- 多维度加权：CER(40%) + 流畅度 (30%) + 相关性 (30%)
- CER 基于 ASR 转录计算字符错误率
- 流畅度基于句子长度分布和 n-gram 重复度
- 相关性基于嵌入相似度
- 与人工标注相关性达 0.82

### Q3: 端到端 Pipeline 的延迟如何优化？
**A**: 
- 流式 ASR：将 ASR 延迟从 10.5s 降至 2s
- 增量 LLM 生成：首 token 延迟降至 0.5s
- 并行处理：Diarization 和部分 ASR 可并行
- 总体从 23.5s 降至 8s（优化 66%）

### Q4: 为什么选 ChromaDB？
**A**: 
- 轻量级，支持持久化存储
- API 简单，易于集成
- 支持元数据过滤（说话人、时间等）
- 适合快速原型和中小规模部署
