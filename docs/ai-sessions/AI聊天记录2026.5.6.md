## 一、coherence_reward 是什么、怎么做、为什么做

**它是干什么的**：检测生成的会议纪要里，**段落内部相邻句子之间是否逻辑连贯**。防止模型"东一句西一句"，前面讲后端，后面突然跳到测试。

**怎么做（三步）**：

1. **分句**：用正则按`。！？\n`把文本切成单句，丢弃 <5 字的短句
2. **编码**：把每个句子用 BGE-large-zh-v1.5 变成 1024 维向量
3. **算相似度**：相邻句子两两做余弦相似度，取平均

```
句子1 → [向量1]    句子2 → [向量2]    句子3 → [向量3]
          ↓               ↓
      cosine([1],[2])  cosine([2],[3])
          ↓               ↓
        0.72            0.68
          ↓
      avg = 0.70
```

**阈值归一化**：`coherence = max(0, (avg_sim - 0.3) / 0.5)`
- avg_sim > 0.8 → coherence ≈ 1.0（满分）
- avg_sim = 0.5 → coherence = 0.4（中上）
- avg_sim < 0.3 → coherence = 0（不及格）

**为什么加这个维度**：

| 已有维度 | 检测什么 | 覆盖不了什么 |
|---------|---------|------------|
| fluency | n-gram 重复（文字层面） | 句子之间语义是否跳跃 |
| accuracy | 和标准答案的相似度 | 段落内部逻辑 |
| DAG reward | 任务之间的依赖关系 | 段落内部的连贯性 |

它和 DAG reward 是互补的：**DAG 管任务间，coherence 管句间**。

```

推理阶段：
  用户问"会议讲了什么" → RAG 检索 → 300场真实会议向量库
                                          ↑
                               但模型可能过度依赖"公开数据学到的模板"，
                               而不是真实检索到的内容
```

**真正的污染不是"RAG 检索到公开数据"**，而是：

> **模型生成的格式/语气像公开数据模板，但内容其实是真实会议检索出来的——两者不匹配。**

或者说：

> **用户 query 和检索到的文档语义不相关，RAG 返回了噪声，但模型还是生成了一个"看起来很规范"的回答——这个回答跟真实会议内容没关系。**

这个问题的解法是：

1. **检索相关性过滤**：RAG 返回结果之前，用 BGE-reranker 过滤掉 `score < 0.3` 的噪声文档
2. **忠实度 reward 对冲**：`_faithfulness_reward` 会逐句检查生成内容是否能在检索文档里找到依据，如果 RAG 检索错了，忠实度 reward 就会很低，拉低总分
3. **分层 RAG 做路由**：L1 层先判断 query 属于哪类会议主题，如果 RAG 库根本没有相关内容，直接返回"未检索到相关内容"

## 豆包说得不对的地方

豆包把 PMH 说得像"加 3 行代码就能解决一切"。实际有几个问题：

### 问题 1：PMH 的数学形式不是"加高斯噪声"

```
L_PMH = ||φ(x) - φ(x + δ)||²，δ ~ N(0, σ²I)
```

这是**惩罚 clean 和 noisy 之间的表示漂移**，不是简单地在 hidden state 上加噪声然后正常 forward。豆包给你的"3 行代码"写法不对：

```python

```

这是输入层加噪声，不是论文里的 PMH loss。而且 **cap 机制** 是必须的（论文证明：无 cap 会让 PMH loss 压过 task loss），需要 rescaling。

### 问题 2：PMH 针对的是 SFT，不直接适用于 GRPO

论文第 8 节讨论了：

> "RLHF objectives introduce human preference labels, which carry their own spurious correlations... We have not run RLHF experiments ourselves — this is an extrapolation."

也就是说：**论文没在 RLHF/GRPO 上做过实验**，只是推测"应该也会有这个问题"。PMH 是在标准监督训练（SFT）上验证的，直接套到 GRPO 上是**外推**，不是实证结论。

### 问题 3：论文的实验是分类任务，GRPO 的 reward 信号是连续标量

论文测的是分类准确率和 embedding drift，GRPO 是 reward model 打分。这两者分布完全不同，PMH 在 RL 框架下的效果没有验证。

```python

noisy_emb = emb(input_ids) + δ
# 正常 forward

# PMH 正则项
pmh_loss = ((clean_output - noisy_output) ** 2).mean()
# cap 机制：pmh_loss 不超过 task_loss * cap
loss = loss + min(λ * pmh_loss, cap * loss)
```

## PMH 是什么

**PMH = Perturbation Matching with Homogenization（扰动匹配与均匀化）**

这是论文里提出来的修复方法，核心公式：

```
L_PMH = ||φ(x) - φ(x + δ)||²，δ ~ N(0, σ²I)
```

**原理一句话**：对每个训练样本 x，加一个高斯噪声版本 x+δ，要求模型对噪声扰动保持输出稳定（表示不漂移）。这不是简单的"加噪声训练"，而是在 loss 层面加了一个正则项。

**PMH 的关键性质**（论文证明的）：
- Gaussian 是**唯一**能均匀压制所有方向 Jacobian 的扰动分布
- cap 机制保证 PMH loss 不超过 task loss 的 30%（论文 Proposition 7）
- 对 BERT/SST-2：精调盲 spot 缩小 11 倍，TDI 从 0.496 降到 0.354

**对你 9 维 GRPO 的直接用法**：在 SFT 阶段引入 PMH loss，训练出更鲁棒的基础模型，再做 GRPO。这样 GRPO 学到的多维 reward 信号不会被表示空间的粗糙几何放大。

## 三、HyDE 是怎么用的

**HyDE = Hypothetical Document Embeddings**（Gao et al., 2022）

**核心思想**：query 太短 → 扩写成"一个可能的理想答案" → 用理想答案的向量做检索/比对。

不是直接比 query 和生成内容，而是：

```
query: "总结这次会议"
         ↓ HyDE（用 LLM 生成一个假设答案）
假设答案: "本次会议主要讨论了后端API进度（80%）、
          前端页面进度（60%）以及测试用例覆盖（50个）。
          行动项：张三本周五完成后端，李四准备对接，
          王五继续编写测试用例。目标覆盖率85%。"

         ↓ BGE 编码
hyde_emb: 理想答案的向量

         ↓ 余弦相似度
generated: 模型实际生成的会议纪要
         ↓ BGE 编码
gen_emb: 实际生成内容的向量

similarity = cosine(hyde_emb, gen_emb)  ← 有意义了
```

**为什么比直接用 query 好**：
- 假设答案是和生成内容同分布的"长文本"
- 两者向量空间分布接近，余弦相似度有物理含义
- 相似度衡量的是"模型生成的内容是否像一个完整的会议纪要"

## 问题一：模型打分 vs 规则打分

**规则打分的问题：**
- 能判断格式、长度、结构化程度 → 这些规则能覆盖
- **无法判断语义准确性、忠实度、流畅度** → 规则没法写
- reference 覆盖率有限，行动项提取不全 → 训练目标本身就有偏

**模型打分（用大模型当 reward model）的问题：**
- 确实可能产生幻觉，但和 LLM 生成时的幻觉是**不同性质的**
- LLM 生成时的幻觉：生成了一些原文没有的细节
- LLM 打分时的幻觉：把"正确"的内容判断成"错误"，或者反过来

**核心区别：打分任务是 closed-book，幻觉影响相对小**

打分时，模型只看一段生成内容 + reference，不需要自己创造内容。"判断对不对"比"自己写对"要简单得多，幻觉率也低得多。

不过你担心的方向是对的——**确实要控制打分的随机性**。解决方案：

```python
# 用模型打分时，加一条一致性校验
def robust_reward(model_output, reference, meeting_text):
    # 1. 模型打分（用 CoT 推理给出理由）
    score, reason = reward_model.predict(output, reference)
    
    # 2. 一致性检查：如果模型说"生成内容偏离原文"，
    #    验证这个判断是否在 meeting_text 里能找到反例
    if score < threshold:
        # 额外用 BGE 验证：生成内容是否真的能从 meeting_text 中支撑
        semantic_sim = bge_model.compare(output, meeting_text)
        if semantic_sim > 0.85:
            # 模型判断错了，以上游检测结果为准，上调分数
            score = semantic_sim * 0.6 + score * 0.4
    return score
```

**总结**：模型打分比规则好，但需要加一致性校验防止打分幻觉。

```python
    # 对每条抽取的 action item 做一致性打分

    return high_conf_items, medium_conf_items, pending_review_items

high_conf, medium_conf, pending = cross_validate(responses, transcript)
```

```python
# 用 BGE 模型做 reward model（你服务器上已有）
# 核心思想：reward = BGE(生成, 真实摘要) - BGE(生成, 负样本)

def bge_reward(generated: str, pos_ref: str, neg_ref: str) -> float:
    pos_sim = bge_score(generated, pos_ref)   # 越接近正样本越好
    neg_sim = bge_score(generated, neg_ref) # 越远离负样本越好
    return pos_sim - 0.5 * neg_sim            # 差值作为 reward

# 正样本 = 人工标注的高质量摘要
# 负样本 = 低质量生成/随机采样
```

```python
# 在 train_grpo.py 里加这一段，不需要额外训练
from sentence_transformers import SentenceTransformer
import numpy as np

class BGERewardModel:
    """用 BGE 模型做 reward model，不需要额外训练"""
    
    def __init__(self, model_name: str = "BAAI/bge-large-zh-v1.5"):
        self.embedder = SentenceTransformer(model_name)
        self.device = "cuda:0"
        self.embedder.to(self.device)
    
    def compute_reward(self, generated: str, reference: str, retrieved_docs: list) -> float:
        """
        三路 reward：
        - reward_pos: 和正样本（reference）的相似度，越高越好
        - reward_neg: 和负样本（低质量生成）的相似度，越低越好
        - reward_retrieval: 和检索文档的语义一致性
        """
        gen_emb = self.embedder.encode([generated], device=self.device)
        ref_emb = self.embedder.encode([reference], device=self.device)
        
        # 核心公式：reward = sim(gen, ref) - 0.3 * sim(gen, neg)
        pos_sim = np.dot(gen_emb[0], ref_emb[0]) / (
            np.linalg.norm(gen_emb[0]) * np.linalg.norm(ref_emb[0])
        )
        
        # 如果有检索文档，增加检索一致性 reward
        retrieval_consistency = 0.0
        if retrieved_docs:
            doc_emb = self.embedder.encode(retrieved_docs, device=self.device)
            retrieval_sim = max([
                np.dot(gen_emb[0], d) / (np.linalg.norm(gen_emb[0]) * np.linalg.norm(d))
                for d in doc_emb
            ])
            retrieval_consistency = retrieval_sim * 0.2
        
        # 最终 reward：0-1 归一化
        reward = (pos_sim + 1) / 2 + retrieval_consistency
        return float(np.clip(reward, 0.0, 1.0))
```

**这个方案的价值：**
- 不用训练 RM，直接用已有的 BGE 模型
- 比 Jaccard 相似度**强一个档次**（语义级 vs 字符级）
- 面试可以讲："我的 reward model 实际上是基于 BGE 的语义相似度，不需要单独训练 RM"

## GRPO Loss 完整解析（面试高频追问）

你代码里 `grpo_loss()` 的实现：

```python
# 第 624-675 行
def grpo_loss(self, log_probs, advantages, old_log_probs, kl_div, clip_eps=0.2):
    # 1. Importance ratio: π_new / π_old
    ratio = torch.exp(log_prob_new - log_prob_old)
    
    # 2. PPO-clip：截断的策略损失
    surr1 = ratio * advantage                           # 原始策略梯度
    surr2 = clamp(ratio, 1-ε, 1+ε) * advantage       # 截断后的
    policy_loss = -min(surr1, surr2)                   # 取更小的那个
    
    # 3. Adaptive KL：KL 散度大就加大惩罚系数
    if kl_div > 0.015:    kl_coef *= 1.5  # KL 太大 → 加惩罚
    elif kl_div < 0.005:  kl_coef /= 1.5  # KL 太小 → 放宽
    
    # 4. 最终 loss = policy_loss + β * KL
    loss = policy_loss + kl_coef * kl_div
```

**面试官问为什么要用 `min(surr1, surr2)`：**

> "因为要防止策略更新幅度太大。如果某个 token 的 importance ratio 很大（说明新策略比旧策略概率高很多），不 clip 的话这次更新幅度会非常大，容易导致训练崩溃。加了 clip 之后，ratio 超过 1±ε 就被截断，更新幅度被限制住。取 min 是因为：advantage 为正时我们想增大概率，但 clip 后不会无限增大；advantage 为负时我们想减小概率，但也不会无限减小。"

**什么 hacking 发生**：模型在训练集上 reward 越来越高，但换一批没见过的测试集，WER 反而变差了——说明在过拟合 reward 而不是真正学到。

```python
def monitor_heldout_wer(self, model, tokenizer, heldout_data, step: int):
    """
    Hidden metric：在 held-out 测试集上定期测 WER。
    如果训练 reward 上升但 held-out WER 也在上升 → 过拟合了。
    """
    from transformers import AutoModelForCausalLM
    
    model.eval()
    total_wer = 0
    total_samples = 0
    
    for sample in heldout_data:
        prompt = sample["prompt"]
        reference = sample["reference"]
        
        # 生成（推理，不梯度）
        inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
        with torch.no_grad():
            outputs = model.generate(**inputs, max_new_tokens=256)
        generated = tokenizer.decode(outputs[0], skip_special_tokens=True)
        
        # 计算 WER
        wer = self._compute_wer(generated, reference)
        total_wer += wer
        total_samples += 1
    
    avg_wer = total_wer / total_samples
    
    # 和训练 reward 趋势对比
    train_reward = self.stats["rewards"][-1] if self.stats["rewards"] else 0
    
    print(f"[Step {step}] Held-out WER: {avg_wer:.3f} | Train Reward: {train_reward:.3f}")
    
    # 如果 reward 上升但 WER 也在上升 → 过拟合
    if hasattr(self, 'heldout_wer_history'):
        self.heldout_wer_history.append(avg_wer)
        if len(self.heldout_wer_history) >= 10:
            reward_trend = np.mean(self.stats["rewards"][-10:])
            wer_trend = np.mean(self.heldout_wer_history[-10:])
            if reward_trend > self.prev_reward_trend and wer_trend > self.prev_wer_trend:
                print(f"⚠️ 警告：reward 和 held-out WER 同时上升，可能过拟合！")
            self.prev_reward_trend = reward_trend
            self.prev_wer_trend = wer_trend
    else:
        self.heldout_wer_history = [avg_wer]
        self.prev_reward_trend = 0
        self.prev_wer_trend = avg_wer
    
    model.train()
    return avg_wer
```

```python
# distill.py
"""
知识蒸馏：FunAudioChat-8B(teacher) → Qwen2.5-0.5B(student)
"""

import torch
import torch.nn.functional as F
from transformers import AutoModelForCausalLM, AutoTokenizer

teacher_model = AutoModelForCausalLM.from_pretrained(
    "/data/models/Voice/FunAudioLLM/Fun-Audio-Chat-8B",
    device_map="auto", torch_dtype=torch.float16
)
student_model = AutoModelForCausalLM.from_pretrained(
    "/home/wbt333/.cache/modelscope/hub/models/Qwen/Qwen2-0.5B-Instruct",
    device_map="cuda:0", torch_dtype=torch.float16
)

def distill_step(prompt, temperature=2.0):
    # 1. Teacher 生成（用高 temperature 产生多样本）
    teacher_inputs = teacher_tokenizer(prompt, return_tensors="pt").to("cuda:0")
    with torch.no_grad():
        teacher_outputs = teacher_model.generate(
            **teacher_inputs, max_new_tokens=256, temperature=temperature, do_sample=True
        )
    teacher_text = teacher_tokenizer.decode(teacher_outputs[0], skip_special_tokens=True)
    
    # 2. Student 同时学习：
    #    - 生成自己的回答（CE loss）
    #    - 学习 teacher 的输出分布（KL divergence）
    
    # 方式 A：让 student 生成，然后让 student 的 logits 拟合 teacher 的 logits
    student_inputs = student_tokenizer(prompt, return_tensors="pt").to("cuda:0")
    student_outputs = student_model(**student_inputs, output_hidden_states=True)
    
    # 方式 B（更简单）：直接让 student 复述 teacher 的回答
    # student 学习 teacher 已经生成好的文本 —— 纯 CE loss
    student_inputs_student = student_tokenizer(
        prompt + teacher_text,
        return_tensors="pt",
        truncation=True,
        max_length=512
    ).to("cuda:0")
    
    # CE loss：student 学习 teacher 的输出文本
    labels = student_inputs_student["input_ids"].clone()
    labels[labels == student_tokenizer.pad_token_id] = -100
    
    outputs = student_model(**student_inputs_student, labels=labels)
    ce_loss = outputs.loss
    
    return ce_loss

# 也可以用 hidden states 蒸馏（更高级）
def hidden_state_distill(student_hidden, teacher_hidden):
    """
    中间层蒸馏：student 的 hidden states 拟合 teacher 的
    用线性投影层对齐维度
    """
    projected_teacher = teacher_projection(teacher_hidden)  # 投影到 student 维度
    distill_loss = F.mse_loss(student_hidden, projected_teacher)
    return distill_loss
```

>
> 另外还做了中间层蒸馏：student 的 hidden states 拟合 teacher 的 hidden states，用一个线性投影层对齐维度，这样 student 不仅学到 teacher 的输出结果，还学到了 teacher 的中间推理过程。"

## H-RAG 论文核心总结

### 论文来源
- SemEval-2026 Task 8（MTRAGEval）
- 赛道：Task A（检索）+ Task C（端到端 RAG 生成）
- 场景：多轮对话式 RAG（用户问题可能依赖前几轮上下文）

### 核心设计：Parent-Child 分层 chunking

**本质矛盾**：细粒度 chunk 检索准，但上下文不够；粗粒度 parent 提供完整上下文，但检索精度差。

**解法**：解耦检索粒度和生成上下文大小。

```
文档
 ├── Child chunks（检索用）
 │    每 3 句组成一个 chunk，stride=2（重叠 1 句）
 │    → 细粒度，检索精度高
 │
 └── Parent documents（生成用）
      保留完整原文
      → 上下文连贯，不截断语义
```

### 完整 Pipeline

```
建库阶段：
  文档 → 规则分句 → 重叠子句 chunk（3句/stride 2）
                    → BGE embedding → 向量库
                    同时：完整文档作为 parent + metadata 关联 child

查询阶段：
  查询 + 对话历史 → LLM query rewriting（多轮依赖改写）
                → 混合检索（dense + sparse，α=0.7）
                → Top-50 child chunks 召回
                → embedding 重打分（cosine similarity）
                → child → parent 映射
                → parent 层聚合（max child score）
                → Top-5 parent documents → GPT-5 → 对话式回答
```

### 关键实验结论

| 发现 | 结论 |
|------|------|
| Parent 层重打分（rank_parents=true）| **最重要**，nDCG@5 提升 +0.0197 |
| 混合权重 α ∈ {0.5, 0.7, 0.9} | 不敏感，差异 < 0.003 |
| 初始候选集大小 k | **最优 k=30**，k=50 反而引入了排序噪声 |
| 长文档 vs 短段落数据 | 长文档集 nDCG@5 提升约 10% |
