> 定向补：百度 Agent 岗位 JD
> 重点覆盖：分布式、高并发、微服务、缓存、消息队列、服务治理

---

## 一、分布式 & 高并发基础

### Q1：高并发问题的本质是什么？

**高并发的本质是资源争用**。当并发请求数超过系统的处理能力时，就会出现响应延迟、连接超时、甚至服务崩溃。

核心指标有三个：**QPS**（每秒请求数）、**RT**（响应时间）、**并发数**，三者关系是 `并发数 = QPS × RT`。比如 QPS=1000，RT=0.5s，并发数就是 500——系统需要同时维持 500 个连接。

解决高并发从两个方向入手：**减少请求量**（缓存、CDN）、**提升吞吐能力**（多实例、负载均衡、异步化）。

---

### Q2：如何设计一个高并发接口？

五步走：

1. **CDN 缓存**：静态资源（图片/CSS/JS）放 CDN，浏览器就近访问，不打到后端
2. **网关层限流**：用令牌桶/滑动窗口算法，超限直接拒绝，返回 429 Too Many Requests
3. **Redis 缓存热点数据**：相同 query 的 RAG 检索结果缓存 5 分钟，避免重复 embedding 推理
4. **异步写 DB**：写请求先发到消息队列（Kafka），立即返回成功，消费者慢慢消费
5. **多实例 + 负载均衡**：Nginx 把请求打到多个应用实例，任意一个挂了不影响整体

---

### Q3：分布式 CAP 定理是什么？

**CAP** 指的是一个分布式系统最多只能同时满足两个特性：

- **C（Consistency）**：一致性——所有节点看到的数据一致，读操作总能得到最新写入
- **A（Availability）**：可用性——每次请求都能在有限时间内返回结果
- **P（Partition Tolerance）**：分区容错——网络分区时系统仍能运行

**现实选择：CP 或 AP**。网络分区不可避免，所以必须在 C 和 A 之间二选一。ZooKeeper / etcd 是 CP 系统，保证一致性；Cassandra / Redis Cluster 是 AP 系统，保证可用性。

你的 RAG 知识库用 ChromaDB 是 CP（向量检索要准确），但 ChromaDB 单机部署所以暂时不涉及这个问题。

---

### Q4：BASE 理论是什么？

**BASE = Basically Available + Soft State + Eventually Consistent**。是对 CAP 中 AP 的补充，用于描述实际工程中的妥协方案。

- **Basically Available**：保证核心功能可用，牺牲非核心（比如大促时关闭评论功能）
- **Soft State**：数据状态可以暂时不一致，比如订单"支付中"状态
- **Eventually Consistent**：数据最终会达到一致，比如微信支付回调可能延迟几秒

类比：快递显示"已发货"（软状态），实际在路上，最终到达（最终一致）。

---

### Q5：分布式 ID 有哪些生成方案？

分布式场景下多个服务各自生成 ID，不能用自增主键（多实例会冲突）。

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 128 位随机数 | 无中心，简单 | 无序、占用空间大 |
| Snowflake | 时间戳 + 机器ID + 序列号 | 有序、可推算时间 | 依赖时钟，机器挂了 |
| 数据库自增 | 单独 ID 生成服务 | 有序、简单 | 单点瓶颈 |
| 雪花变体（百度 UidGenerator） | 优化时间回拨问题 | 抗时钟漂移 | 略复杂 |

面试重点说 Snowflake：64 位 = 1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号，能扛 69 年不重复。

## 二、缓存（Redis）

### Q6：Redis 有哪些数据结构？各自适用场景？

| 结构 | 底层实现 | 适用场景 |
|------|----------|----------|
| String | 动态字符串 / int | 验证码、token、简单计数 |
| Hash | 字典 | 用户画像、对象缓存（字段可单独更新） |
| List | 双向链表 / 压缩列表 | 消息队列（BRPOP）、最新 N 条记录 |
| Set | 哈希表 / 整数集合 | 标签、去重（spike 用户 ID 去重） |
| Sorted Set | 跳表 + 字典 | 排行榜、时间线排序（带过期时间） |
| Bitmap | 位数组 | 日活统计、签到、去重计数 |

你的 RAG query 缓存可以用 String（key=query_hash，value=检索结果 JSON，过期时间 5 分钟）。

---

### Q7：Redis 缓存穿透、击穿、雪崩是什么？

这是 Redis 三连问，极其高频。

**缓存穿透**：请求的数据既不在缓存也不在 DB，比如恶意刷不存在的用户 ID。
→ 解决：布隆过滤器（判断 key 是否存在）、缓存空值（NULL 并设置短过期时间）。

**缓存击穿**：热点 key 过期瞬间，大量请求同时打穿到 DB（NBA 总决赛"詹姆斯"关键词）。
→ 解决：互斥锁（只有一个线程查 DB）、永不过期+异步更新（后台定时续命）。

**缓存雪崩**：大量 key 同时过期，或者 Redis 宕机，所有请求打到 DB。
→ 解决：过期时间加随机偏移量（key1@600s、key2@650s）、Redis 集群高可用（主从+哨兵）、服务熔断降级。

---

### Q8：Redis 缓存和 MySQL 如何保证一致性？

**Cache Aside 模式**（最常用）：
- 读：先查 Redis，没有则查 MySQL，然后把结果写入 Redis
- 写：先更新 MySQL，再删除 Redis 中的缓存（**是删除不是更新**，防止并发时脏数据）

为什么是删除不是更新？因为更新缓存时，如果另一个请求同时更新了 DB 并写入了不同值，并发会导致缓存是旧值。**删除缓存**之后，下次读会从 DB 取到最新值。

**强一致场景**：用延迟双删——先删缓存、更新 DB、等待 N 毫秒再删一次（处理主从延迟期间产生的脏缓存）。

---

### Q9：Redis 的持久化机制是什么？

**RDB（快照）**：定时 fork 子进程，把内存数据快照写入磁盘。恢复快，但可能丢最近的数据。
**AOF（日志）**：每个写命令追加到文件，恢复时重放命令。数据完整，但文件大、恢复慢。
**混合持久化**（Redis 4.0+）：RDB 做全量 + AOF 做增量，兼顾恢复速度和完整性。

生产环境一般开 AOF + RDB，RDB 做定时全量备份，AOF 保证实时性。

---

### Q10：Redis 集群是如何工作的？

**主从复制**：一主多从，主节点写，从节点读（读写分离）。主挂了需要手动切换。
**哨兵（Sentinel）**：自动监控主从节点，主挂了自动投票选新主，更新客户端配置。
**Cluster 模式**（官方集群）：数据分 16384 个槽，每个节点负责一部分槽，自动扩缩容。客户端请求 key 时，根据 CRC16(key) % 16384 算出槽，直接打到对应节点（Smart 客户端）或通过代理（Proxy）。

你的 RAG 缓存如果要做高可用，可以用 Redis Sentinel，主节点挂了哨兵自动切换。

## 三、消息队列

### Q11：为什么需要消息队列？

三个核心价值：

1. **异步解耦**：上传音频后立即返回 task_id，RAG 检索和 LLM 生成异步处理，响应时间从 5s 降到 0.5s
2. **削峰填谷**：瞬时流量洪峰（比如 1000 人同时上传），消息队列作为缓冲，消费者按自己节奏消费
3. **系统解耦**：生产者和消费者不直接依赖，任意一方挂了不影响另一方

类比：双十一零点洪峰 -> 订单系统先收请求 -> 丢进 MQ -> 后台慢慢处理 -> 用户等"订单正在处理"而不是页面崩溃。

---

### Q12：Kafka 和 RabbitMQ 的区别是什么？

| 维度 | Kafka | RabbitMQ |
|------|-------|----------|
| 架构 | 分区 + 副本 | 交换机 + 队列 + 绑定 |
| 吞吐量 | 百万级 QPS | 万级 QPS |
| 消息顺序 | 分区内有序 | 单队列有序，多队列无序 |
| 消息持久化 | 顺序写磁盘，可回溯 | 可持久化，不支持回溯 |
| 延迟 | 毫秒级 | 微秒级（延迟更低） |
| 适用场景 | 日志采集、大数据流处理 | 业务消息、任务队列 |

**选型原则**：大数据/流处理选 Kafka，低延迟业务队列选 RabbitMQ。你的音频处理流水线用 Kafka 更合适（吞吐高、支持回溯重消费）。

---

### Q13：消息队列如何保证不丢消息？

三个环节，每个都要做：

**生产者端**：
- Kafka：开启 `acks=all`（所有 ISR 副本都写入才确认）+ `producer.type=sync` + 失败重试
- RabbitMQ：开启 publisher confirm 机制

**Broker 端**：
- Kafka：`replication.factor >= 3`（3 副本）+ `min.insync.replicas >= 2`（至少 2 个副本写入成功）+ 刷盘策略 `fsync`
- RabbitMQ：队列持久化 `durable=true` + 消息持久化 `delivery_mode=2`

**消费者端**：
- 关闭 auto ack，手动 ACK 成功后才删除消息
- 业务处理完成再 ACK，如果处理失败 NACK 回队列重试

---

### Q14：消息队列的消费幂等性如何保证？

消息队列有**at least once**（至少一次，可能重复）和 **exactly once**（精确一次）两种语义。Kafka 支持 exactly once（事务 API），但大多数场景做到 **at least once + 业务幂等** 就够了。

幂等实现方案：
1. **数据库唯一索引**：消费前先 INSERT，重复插入会报错（利用 DB 约束天然幂等）
2. **Redis 去重**：消费前查 Redis 是否处理过（key=message_id，过期时间=消息生命周期）
3. **版本号机制**：每条消息带版本号，更新时 `WHERE version = new_version`，版本对不上就不更新

你的场景：RAG 检索结果写入可以用消息 ID 做幂等，避免重复存储。

## 四、微服务 & 服务治理

### Q15：微服务拆分原则是什么？

两个核心原则：

**康威定律**：系统架构是组织结构的镜像——一个团队不应该维护超过 5 个服务（亚马逊 two-pizza team）。如果两个服务总是同时上线发布，说明它们应该合并。

**领域驱动设计（DDD）**：按业务边界拆分，不是按技术层次。订单域、支付域、用户域各自独立，域之间通过 API 通信。

你的系统可以这样拆：
- ASR 服务（音频转文字）
- RAG 服务（向量检索）
- LLM 服务（大模型推理）
- 调度服务（任务编排）

---

### Q16：服务注册与发现是如何工作的？

**注册中心**（Consul / Nacos / etcd）：所有服务实例启动时把自己的 IP+端口注册进去，关闭时注销。

**客户端发现**（Ribbon）：消费者从注册中心拉取服务列表，自己做负载均衡。
**服务端发现**（Nginx / Gateway）：消费者请求网关，网关从注册中心拿列表并做负载均衡，客户端无感知。

> "注册中心心跳机制：每个服务每隔 5 秒向注册中心发心跳，连续 3 次没收到就认为挂了，从列表里摘除。这解决了服务动态扩缩容的问题——新增实例自动注册，挂了自动下线。"

---

### Q17：熔断和限流是什么？

**限流**：系统入口控制请求速率，保护下游。
- 计数器算法：每秒 N 个请求，超限拒绝
- 滑动窗口：比计数器精确，窗口内计数而不是整秒
- 令牌桶：以固定速率放令牌，请求来了拿令牌，拿不到就拒绝（允许突发）

**熔断**：当下游服务频繁超时/报错时，主动短路，不再调用下游，直接返回降级结果。
- 类比：电路保险丝，过流就跳闸
- 恢复：熔断后每 N 秒放一个请求试试，能通就逐步恢复

Sentinel / Hystrix 是常用的熔断框架。

---

### Q18：链路追踪是什么？为什么需要？

一个用户请求经过 10 个微服务，哪个环节慢？日志散落在各个机器，查起来非常困难。

**链路追踪**：每个请求分配一个 Trace ID，在 HTTP Header 里透传，每个服务记录"自己处理了多长时间"。Jaeger / SkyWalking 把所有片段串联起来，生成一张调用链路图。

面试话术：
> "我的音频处理流水线有 ASR→RAG→LLM 三个阶段，如果线上用户反馈慢，我可以 Trace ID 定位是 RAG 检索慢还是 LLM 推理慢，精确到毫秒级别。"

## 五、数据库

### Q19：索引失效的常见原因有哪些？

| 场景 | 原因 |
|------|------|
| `%abc%` | like 模糊查询以 % 开头，无法用 B+ 树 |
| `age + 1 = 30` | 对索引列做了计算 |
| `WHERE a = 1 OR b = 2` | OR 条件有一边没索引，全表扫描 |
| `WHERE LEFT(name, 4) = '张'` | 函数作用于索引列 |
| 隐式类型转换 | `WHERE phone = 1380013800`（phone 是 VARCHAR） |
| `!=` / `NOT IN` | 不确定范围，通常不走索引 |

---

### Q20：MySQL 的事务隔离级别是什么？

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|----------|------|----------|------|----------|
| Read Uncommitted | 可能 | 可能 | 可能 | 无锁 |
| Read Committed | 不可能 | 可能 | 可能 | MVCC 快照 |
| Repeatable Read（默认） | 不可能 | 不可能 | 可能 | MVCC + GAP 锁（InnoDB） |
| Serializable | 不可能 | 不可能 | 不可能 | 锁表 |

> "RR（Repeatable Read）是 MySQL InnoDB 的默认级别。MVCC 的核心是每个事务有自己的 Read View 快照，开事务时的数据版本对本次事务一直可见，解决了不可重复读。幻读靠 GAP 锁解决——锁住行之间的间隙，防止新插入的行被读到。"

## 六、综合设计题

### Q21：如何设计一个抢红包系统？

高频系统设计题。

1. **入库时唯一键防重复**：同一个用户 ID + 红包 ID 只能有一条领取记录，重复请求直接报错（利用 MySQL 唯一索引）
2. **Redis 预减库存**：红包总量放 Redis，请求进来先 `DECR`，负数直接返回失败，比 DB 操作快 10 倍
3. **MQ 异步落库**：扣减成功后发 MQ，消费者异步写 MySQL，避免 DB 被打爆
4. **乐观锁更新**：UPDATE hongbao SET remain = remain - $amount WHERE id = ? AND remain >= $amount
5. **分段下发**：不是一次性生成所有红包，分段预热，减少 Redis 压力

## 七、快速术语表

| 术语 | 一句话解释 |
|------|-----------|
| 幂等 | 同一个操作执行多次结果一样 |
| 雪花算法 | 分布式 ID 方案，时间戳+机器+序列号 |
| 读写分离 | 主库写，从库读，分担压力 |
| 分库分表 | 数据量大了按字段拆到多个库/表 |
| 异地多活 | 多个机房同时提供服务，一个挂了另一个接管 |
| 灰度发布 | 新版本只给小部分用户用，逐步放量 |
| 热key问题 | 某个 key 被大量请求打爆（明星塌房） |
| 幂等设计 | 并发/重试下结果一致 |

## 八、面试话术模板

遇到不会的题，这样说：

> "这部分我了解不深，但我的理解是……如果遇到这个场景，我会这样处理……"

比如问到"分库分表"：

> "分库分表我了解不多，主要用过的是按 user_id 哈希分片，用 ShardingSphere 做中间件。不过我的 RAG 项目目前数据量还没到需要分库分表的程度，用 ChromaDB 单机存储就够了，如果未来数据量增长到千万级，我会考虑按时间分表，冷热数据分离。"

这样回答既诚实，又展示你知道正确的方向，还主动把话题引回你的项目。
