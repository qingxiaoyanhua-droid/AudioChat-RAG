- 为后续 GRPO 强化学习打下基础

### 4.2 损失函数（核心！必考手撕）

**文件位置**：`train_sft.py` + `sft_loss_function.py`

**SFT Loss 公式**：
```
Loss = -Σ y_true × log(y_pred)
     = -log(p(next_token | context))
```

**关键优化：Response-only Loss Masking（面试必问）**

只对 **assistant 回复部分** 计算 loss，prompt 部分 label 设为 -100（PyTorch 自动忽略）。

**为什么要这样做？**
- 如果对 prompt 也算 loss，模型学的是"如何复读 prompt"而不是"如何回答"
- Response-only masking 让模型专注于学习生成格式化的会议纪要

**代码实现**：
```python
def format_conversation(sample, tokenizer, max_length):
    conversation = sample["conversation"]
    
    # 1. 分别 tokenize prompt-only，确定边界
    prompt_only = [m for m in conversation if m["role"] != "assistant"]
    prompt_text = tokenizer.apply_chat_template(
        prompt_only, tokenize=False, add_generation_prompt=True
    )
    prompt_ids = tokenizer(prompt_text)["input_ids"]
    prompt_len = len(prompt_ids)
    
    # 2. Tokenize 完整对话
    full_text = tokenizer.apply_chat_template(conversation, tokenize=False)
    tokenized = tokenizer(full_text, truncation=True, max_length=max_length)
    input_ids = tokenized["input_ids"]
    
    # 3. Response-only masking: prompt 部分 label = -100
    labels = [-100] * prompt_len + input_ids[prompt_len:]
    
    return {"input_ids": input_ids, "attention_mask": tokenized["attention_mask"], "labels": labels}
```

**手撕代码**（面试可能会让写）：
```python
def sft_loss(logits, labels):
    """
    logits: [batch_size, seq_len, vocab_size]
    labels: [batch_size, seq_len]
    """
    # 1. 计算 log_softmax
    log_probs = torch.log_softmax(logits, dim=-1)  # [B, S, V]
    
    # 2. 获取正确 token 的 log 概率
    # labels.unsqueeze(-1): [B, S, 1]
    correct_log_probs = log_probs.gather(-1, labels.unsqueeze(-1)).squeeze(-1)  # [B, S]
    
    # 3. 取平均并加负号
    loss = -correct_log_probs.mean()
    
    return loss
```

### 5.2 RAG 流程详解（必背）

**文件位置**：`audiochat/rag/retriever.py` + `audiochat/rag/storage.py`

**离线建库**：
```python
# 会议记录入库（setup_rag_db.py）
def add_meeting_record(meeting_id, utterances):
    for u in utterances:
        doc = MeetingDocument(
            content=f"{u.speaker}: {u.text}",
            meeting_id=meeting_id,
            speaker=u.speaker,
            timestamp=u.timestamp
        )
        embedding = embedder.encode(doc.content)  # BGE → 1024 维向量
        storage.add(doc, embedding)                # 写入 ChromaDB
```

**在线检索（当前实现）**：
```python
def retrieve(query, k=3, recall_k=15):
    # Stage 1: Bi-Encoder 粗排 + 时间衰减
    docs = storage.search(query, k=recall_k)  # ChromaDB 余弦相似度 → Top-15
    for doc in docs:
        decay = time_decay(doc.timestamp)      # 0.5^(days/7)
        if is_exempt(doc): decay = 1.0         # 关键词豁免
        doc.score = dense_score * decay

    # Stage 2: Cross-Encoder 精排（全部候选送入，不截断）
    rerank_scores = reranker.predict([(query, doc) for doc in docs])

    # Stage 3: Soft decay 最终打分
    for doc in docs:
        doc.final = rerank_score * decay**0.5  # sqrt 拉平衰减曲线

    return sorted(docs, key=final)[:k]  # Top-3
```

```python
def build_rag_prompt(query, contexts):

【当前问题】{query}
```

**时间衰减公式**（必考手撕）：
```python
def time_decay(timestamp, half_life_days=7.0):
    """
    指数时间衰减
    decay = 0.5 ^ (days_diff / half_life)
    """
    doc_time = datetime.fromisoformat(timestamp)
    now = datetime.now()
    days_diff = (now - doc_time).days
    
    decay = 0.5 ** (days_diff / half_life_days)
    return max(0.1, decay)  # 最小 0.1，避免完全忽略旧数据
```

### 5.4 两阶段检索：BGE Embedding 粗排 + BGE Reranker 精排（核心亮点）

**为什么分粗排和精排？（面试必问）**
- **Bi-Encoder（粗排）**：query 和 doc **分别**编码成向量，余弦相似度匹配，速度快（O(1) 查向量库），适合万级文档召回 Top-15
- **Cross-Encoder（精排）**：query 和 doc **拼接**后输入 Transformer，**交叉注意力**计算相关性分数，精度远高于 Bi-Encoder，但速度慢
- 这是工业界标准的 **recall → rerank** 两阶段架构（和搜索引擎的粗排/精排一脉相承）

**完整检索流程**（`audiochat/rag/retriever.py`）：
```python
# Stage 1: BGE Embedding (Bi-Encoder) 粗排 → Top-15 + 时间衰减
docs = storage.search(query, k=15)          # ChromaDB 余弦相似度
for doc in docs:
    decay = 0.5 ** (days_diff / 7.0)        # 半衰期 7 天
    if is_exempt(doc): decay = 1.0          # 架构/决策类内容豁免
    doc.score = dense_score * decay

# Stage 2: BGE Reranker (Cross-Encoder) 精排
# 全部 Top-15 候选送入，不截断
pairs = [(query, doc.content) for doc in docs]
rerank_scores = reranker.predict(pairs)     # 交叉注意力打分

# Stage 3: Soft decay 最终打分
for doc in docs:
    doc.final = rerank_score * decay**0.5   # sqrt 拉平衰减曲线
# 排序取 Top-3
```

**Bi-Encoder vs Cross-Encoder 对比（面试高频）**：
| 特性 | BGE Embedding (Bi-Encoder) | BGE Reranker (Cross-Encoder) |
|------|---------------------------|------------------------------|
| 编码方式 | query/doc **分别**编码 | query+doc **拼接**后编码 |
| 注意力 | 无交互 | **交叉注意力（Cross Attention）** |
| 输出 | 1024 维向量 → 余弦相似度 | 直接输出 0-1 相关性分数 |
| 速度 | 快（可预计算 doc 向量） | 慢（每对都要重新计算） |
| 精度 | 中 | **高** |
| 适用 | 粗排 / 召回（万级→Top-15） | 精排（Top-15→Top-3） |

**代码对比**（面试可写）：
```python
# Bi-Encoder：分别编码，速度快
query_emb = embedder.encode(query)    # [1024]
doc_emb = embedder.encode(doc)        # [1024]
sim = cosine_similarity(query_emb, doc_emb)

# Cross-Encoder：拼接输入，精度高
score = reranker.predict([(query, doc)])  # 直接输出 0-1 分数
```

**消融实验结果**：
| 检索配置 | Top-3 准确率 | BGE Sim |
|----------|-------------|---------|
| Bi-Encoder Only | 76% | 0.79 |
| Bi-Encoder + 时间衰减 | 79% | 0.81 |
| **Bi-Encoder + 时间衰减 + Reranker** | **83%** | **0.84** |

### 5.5 时间衰减的缺陷与解决（一面被追问过！）

**面试官的问题**："如果重要内容在很早的会议记录里，时间衰减把分数拉得很低怎么办？"

**问题本质**：直接乘 decay 太猛。30 天前的文档 decay = 0.5^(30/7) ≈ 0.05，即使 Reranker 给了 0.9 的高分，最终 0.9 × 0.05 = 0.045，几乎被淘汰。

**核心方案：Soft Decay — 用 sqrt(decay) 代替 decay**

```
30 天前的文档，half_life=7 天：
  decay = 0.5^(30/7) ≈ 0.05

  硬衰减：rerank 0.9 × 0.05 = 0.045  ← 几乎淘汰
  软衰减：rerank 0.9 × √0.05 = 0.9 × 0.22 = 0.20  ← 仍有竞争力

原理：开根号等效于把半衰期翻倍（7天→14天），
      让时间做微调而非主导排序。
```

**实现**：
```python
# Reranker 精排后的最终打分
final_score = rerank_score * decay**0.5  # sqrt 拉平衰减曲线
```

**辅助机制：关键词豁免（覆盖确定性场景）**
```python
EXEMPT_PATTERNS = {
    "最终决定", "最终方案", "结论：", "决定采用",
    "选定", "已确定", "拍板", "会议决议", "已批准",
    "确认采用", "批准使用", "结论是", "决定是",
}
# 只有决策已定的结论性文档才豁免——"讨论架构" ≠ "架构已定"
if any(pat in content for pat in EXEMPT_PATTERNS):
    decay = 1.0  # 完全不衰减
```

**面试回答话术**：
> "我遇到过这个问题。直接乘时间衰减太猛——30 天前的文档 decay 只有 0.05，即使 Reranker 打了 0.9 的高分，最终只有 0.045，基本被淹没了。
>
> 我的解决是 **Soft Decay**：最终打分用 rerank_score × sqrt(decay) 而不是 rerank_score × decay。开根号等效于把半衰期从 7 天拉长到 14 天，30 天前的文档从乘 0.05 变成乘 0.22，保留了竞争力。
>
> 同时加了一个**结论性文档豁免**：只有包含"最终决定""结论：""拍板"等结论性短语的内容才不衰减。相比之前用 topic 关键词（"架构""决策"），这种方法更精确——"讨论架构问题"≠"架构已定"，只有决策真正拍板了才豁免。
>
> 半衰期 7 天是根据团队周会节奏定的：每周一有总结会议，信息以周为单位更新，7 天足够让旧信息自然衰减。"

|------|------|------|----------|

## 六、GRPO 强化学习阶段（train_grpo.py）

### 6.1 出发点

**为什么需要 GRPO**：
- SFT 只能教会模型格式，不能优化生成质量
- 需要强化学习来优化：准确性、流畅度、相关性

**得到了什么**：
- 生成质量提升 15%（综合得分 0.76）
- 模型学会生成更准确、流畅、相关的会议纪要

### 6.2 GRPO 核心原理（必背⭐⭐⭐）

**GRPO vs PPO**：
| 特性 | PPO | GRPO |
|------|-----|------|
| Critic 模型 | 需要 Value 网络 | **不需要** |
| 优势计算 | Value 估计 | **组内相对优势** |
| 实现复杂度 | 高 | **低** |
| 计算效率 | 较低 | **较高** |

**GRPO 核心公式**（必考手撕）：
```python
# 1. 组内相对优势（GRPO 核心创新）
A_i = (r_i - mean(r)) / std(r)

# 2. GRPO 损失
L = -E[log(π_new/π_old) × A_i] + kl_coef × KL(π_new || π_ref)
```

### 6.3 奖励函数（grpo_reward_function.py）

**文件位置**：`grpo_reward_function.py` 第 23 行

**多维度奖励公式**（必背）：
```python
reward = 0.40 * accuracy + 0.25 * fluency + 0.15 * relevance + 0.10 * structure + 0.10 * length
```

**模型配置**：
```python
# BGE Embedding（Bi-Encoder）- 粗排，速度快
```

**代码实现**：
```python
class GRPORewardFunction:
    def __init__(self, embedder_model="BAAI/bge-large-zh-v1.5",
                 reranker_model="BAAI/bge-reranker-large"):
        # 加载 BGE 嵌入模型（Bi-Encoder，用于计算语义相似度）
        self.embedder = SentenceTransformer(embedder_model)
        
        # 加载 BGE Reranker（Cross-Encoder，用于精排，当前未使用）
        self.reranker = CrossEncoder(reranker_model)
    
    def compute_reward(self, generated, reference, query, retrieved_docs, action_items=None):
        # 1. 准确性奖励（20%）- BGE Embedding + 余弦相似度
        accuracy_reward = self._accuracy_reward(generated, reference)

        # 2. 忠实度奖励（15%）- 逐句追溯源文档，防幻觉
        faithfulness_reward = self._faithfulness_reward(generated, retrieved_docs)

        # 3. 流畅度奖励（15%）- N-gram 重复度
        fluency_reward = self._fluency_reward(generated)

        # 4. 相关性奖励（8%）- Query 语义相似度
        relevance_reward = self._relevance_reward(generated, query)

        # 5. 结构奖励（10%）- Markdown 格式检查 + owner 合规（合并了 format_reward）
        structure_reward = self._structure_reward(generated, action_items)

        # 6. 长度奖励（10%）- 长度是否合适
        length_reward = self._length_reward(generated)

        # 7. dependency_reward（7%）- DAG 无环验证（对应 Resolve Rate 前置）
        dependency_reward = self._dependency_validity_reward(action_items)

        # 8. executability（7%）- 任务节点可执行（对应 Resolve Rate）
        executability = self._task_executability(action_items)

        # 加权求和
        reward = (0.22 * accuracy_reward + 0.17 * faithfulness_reward +
                  0.17 * fluency_reward + 0.10 * relevance_reward +
                  0.10 * structure_reward + 0.10 * length_reward +
                  0.07 * dependency_reward + 0.07 * executability)

        return reward
    
    def _accuracy_reward(self, generated, reference):
        """准确性：BGE Embedding + 余弦相似度"""
        gen_emb = self.embedder.encode(generated)
        ref_emb = self.embedder.encode(reference)
        sim = cosine_similarity(gen_emb, ref_emb)
        return (sim + 1) / 2  # 归一化到 0-1
    
    def _fluency_reward(self, generated):
        """流畅度：N-gram 重复度（重复越少越流畅）"""
        ngrams = [generated[i:i+3] for i in range(len(generated)-2)]
        if not ngrams:
            return 0.0
        unique_ratio = len(set(ngrams)) / len(ngrams)
        
        # 长度惩罚
        length = len(generated)
        if length < 50 or length > 500:
            length_penalty = 0.5
        else:
            length_penalty = 1.0
        
        return unique_ratio * length_penalty
    
    def _relevance_reward(self, generated, query):
        """相关性/切题：BGE Embedding + 余弦相似度"""
        gen_emb = self.embedder.encode(generated)
        query_emb = self.embedder.encode(query)
        sim = cosine_similarity(gen_emb, query_emb)
        return (sim + 1) / 2
    
    def _structure_reward(self, generated):
        """结构化：Markdown 格式检查"""
        score = 0.0
        if "## " in generated:
            score += 0.3
        if re.search(r'^\d+\.', generated, re.MULTILINE):
            score += 0.3
        if "**" in generated:
            score += 0.2
        return score
    
    def _length_reward(self, generated):
        """长度：是否在合理范围（100-300 字）"""
        length = len(generated)
        if 100 <= length <= 300:
            return 1.0
        elif 50 <= length < 100 or 300 < length <= 500:
            return 0.7
        else:
            return 0.3
```

### 6.4 GRPO 损失函数（train_grpo.py）

**文件位置**：`train_grpo.py` 第 624-655 行

**代码实现**：
```python
def grpo_loss(self, log_probs, advantages, old_log_probs, kl_div):
    """
    GRPO 损失函数
    
    L = -E[log(π_new/π_old) × A_i] + kl_coef × KL(π_new || π_ref)
    """
    # 1. Importance ratio: π_new / π_old
    ratios = []
    for lp, olp in zip(log_probs, old_log_probs):
        ratio = torch.exp(lp - olp)
        ratios.append(ratio)
    
    # 2. Policy loss: -ratio × advantage
    policy_losses = []
    for ratio, adv in zip(ratios, advantages):
        policy_loss = -ratio * adv
        policy_losses.append(policy_loss)
    
    policy_loss = torch.stack(policy_losses).mean()
    
    # 3. KL penalty
    kl_loss = kl_div
    
    # 4. Total loss
    loss = policy_loss + self.args.kl_coef * kl_loss
    
    return loss
```

**训练步骤**（`train_step` 第 521-590 行）：
```python
def train_step(self, batch):
    # 1. 对同一 prompt 生成多个 response（group_size=4）
    response_ids = self.generate_responses(prompt_ids, num_generations=4)
    
    # 2. 计算每个 response 的奖励
    rewards = []
    for resp in response_ids:
        reward = self.reward_fn.compute_reward(resp, reference, query)
        rewards.append(reward)
    
    # 3. 计算组内优势（核心！）
    advantages = self.compute_advantage(rewards)
    # A_i = (r_i - mean(r)) / std(r)
    
    # 4. 计算 log 概率
    log_probs = self.compute_log_probs(prompt_ids, response_ids)
    
    # 5. 计算参考模型 log 概率（用于 importance sampling）
    old_log_probs = self.compute_log_probs_with_model(ref_model, prompt_ids, response_ids)
    
    # 6. 计算 KL 散度
    kl_div = self.compute_kl_divergence(prompt_ids, response_ids)
    
    # 7. 计算 GRPO 损失
    loss = self.grpo_loss(log_probs, advantages, old_log_probs, kl_div)
    
    # 8. 反向传播
    self.optimizer.zero_grad()
    loss.backward()
    torch.nn.utils.clip_grad_norm_(self.model.parameters(), max_norm=1.0)
    self.optimizer.step()
    
    return {"loss": loss.item(), "reward": np.mean(rewards)}
```

**优势计算代码**（必考手撕）：
```python
def compute_advantage(self, rewards):
    """
    组内相对优势计算（GRPO 核心）
    A_i = (r_i - mean(r)) / std(r)
    """
    rewards = torch.tensor(rewards, dtype=torch.float32)
    mean_reward = rewards.mean()
    std_reward = rewards.std() + 1e-8  # 防止除 0
    advantages = (rewards - mean_reward) / std_reward
    return advantages
```

## 七、强化学习八股文（必背⭐⭐⭐）

### Q1: GRPO 和 PPO 的区别？

**回答**：
> "两者最大的区别在**优势计算方式**：
>
> PPO 需要一个 Critic 模型来估计 Value 函数 V(s)，然后 A = R - V(s)。Critic 模型和 Policy 模型一样大，训练成本翻倍。
>
> GRPO 的核心创新是**组内相对优势**：对同一 prompt 生成 G 个 response（我设的 G=4），组内归一化：
> ```
> A_i = (r_i - mean(r)) / std(r)
> ```
> 这样就不需要 Critic 模型了。
>
> **但我保留了 PPO 的 clip 机制**——对 importance ratio 做截断：
> ```
> L = -min(ratio * A, clip(ratio, 1-ε, 1+ε) * A)
> ```
> 原因：裸 ratio 可能很大导致策略跳崩。我消融实验验证了 clip ε=0.2 最稳定，无 clip 时 KL 散度涨到 0.18 且训练震荡。
>
> 另外我实现了**自适应 KL 系数**（Lagrangian style）：KL > 目标×1.5 时自动加大惩罚，KL < 目标×0.5 时自动减小，让训练自动平衡探索和稳定。"

### Q2: 为什么用 GRPO 不用 DPO？

**回答**：
> "三个原因：
>
> 1. **数据成本**：DPO 需要 chosen/rejected 偏好对，构建成本高。GRPO 只需设计奖励函数。
> 2. **灵活性**：我的奖励函数有 6 个维度（准确性/忠实度/流畅度/相关性/结构化/长度），GRPO 可以动态调权重；DPO 只有一个偏好信号。
> 3. **消融实验验证**：我对比了 SFT+DPO vs SFT+GRPO，GRPO 的 ROUGE-L 高 5 个点（0.58 vs 0.53），幻觉率低 6%（18% vs 24%）。
>
> DPO 本质上是把 reward modeling 隐式化了，但在会议纪要场景中，奖励函数可以明确定义，GRPO 更合适。"

### Q3: 奖励函数如何设计？

**回答**：
> "我设计了**八维度**加权奖励（借鉴 Chat2Workflow Resolve Rate 评估思想，新增 dependency/executability）：
> ```
> reward = (0.22*准确性 + 0.17*忠实度 + 0.17*流畅度 + 0.10*相关性
>          + 0.10*结构化 + 0.10*长度 + 0.07*dep + 0.07*exec)
> ```
>
> - **准确性（22%）**：生成内容与 reference 的 BGE 嵌入余弦相似度
> - **忠实度（17%）**：**逐句检查生成内容是否可追溯到源文档**——每个句子与 RAG 检索到的文档计算最大余弦相似度，低于 0.65 阈值的视为幻觉。这是为了**防止 LLM 编造不存在的进度或人名**
> - **流畅度（17%）**：3-gram 唯一比例 × 长度惩罚
> - **相关性（10%）**：生成内容与 query 的语义相似度（切题程度）
> - **结构化（10%）**：Markdown 格式检查（标题/列表/关键词）+ action_items owner 非空检查
> - **长度（10%）**：输出长度是否在合理范围
> - **dependency_reward（7%）**：Kahn 拓扑排序验证 DAG 无环——Kahn 算法核心：统计每个节点入度，删掉入度为0的节点，重复直到队列空，若剩节点则为有环。
>
>   **循环依赖低分例子**：
>   - 任务0说"依赖任务1"，任务1说"依赖任务0"（互相等待，整条工作流卡死）
>   - 三角循环：0→1→2→0
>
>   **DAG 无环高分例子**：
>   - 0无依赖 → 1依赖0 → 2依赖1（单向链）
>   - 0无依赖，1无依赖，2依赖0和1（两个起点并行）
>
> - **executability（7%）**：逐任务检查四项——负责人非空、task长度>=5、有时间关键词、依赖索引有效。四项各0.25分。
>
>   **executability 低分例子**：
>   - owner="待定"，task="干活"，deadline=""（四项全空或模糊）
>   - task="处理"（太笼统，小于5字符）
>   - deadline="某个时间"（没有具体时间关键词）
>   - depends_on=[5]，但总共只有3个任务（第5个不存在）
>
>   **高分例子**：
>   - owner: "张三"，task: "完成后端API联调"，deadline: "本周五"，deps: [] → 四项全满足
>
> Chat2Workflow 启发：格式正确（Pass Rate）≠ 能实际执行（Resolve Rate），两个新增维度分别对应拓扑依赖验证和节点可执行性。format_reward 已合并到 structure_reward 中。消融实验显示新增两维度各贡献约 2 分。"

### Q3.5: 忠实度和准确性有什么区别？（面试易混淆）

**回答**：
> "准确性是和 **reference（标准答案）** 比，衡量'生成的好不好'。
>
> 忠实度是和 **source documents（RAG 检索到的原文）** 比，衡量'生成的内容有没有根据'。
>
> 举例：reference 说'后端完成 80%'，模型生成'后端完成 90%'——准确性高（因为语义接近），但忠实度低（因为源文档里是 80% 不是 90%）。
>
> 这就是为什么需要两个独立的维度。"

### Q4: GRPO 训练不稳定怎么办？

**回答**：
> "我实际遇到过训练震荡，解决方案按有效程度排序：
>
> 1. **PPO-clip（最有效）**：对 importance ratio 截断到 [1-ε, 1+ε]，ε=0.2。消融实验：无 clip reward=0.74，有 clip reward=0.78
> 2. **自适应 KL 惩罚**：Lagrangian style 自动调整 kl_coef。KL 太大时自动加大惩罚，KL 太小时放松，比手动调参稳定
> 3. **Gradient clipping**：max_norm=1.0，防止梯度爆炸
> 4. **Warmup + Cosine Decay**：前 10% 步数 warmup
> 5. **增加 group_size**：4 → 6，更多样本估计优势更准
>
> 最后 KL 散度稳定在 0.12（阈值 0.5 内），clip 比例从 0.08 升到 0.18 再降到 0.14（策略从探索到收敛）。"

### Q5: Importance Sampling 和 PPO-clip 的关系？

**回答**：
> "Importance Sampling 用 ratio = π_new/π_old 修正新旧策略的分布差异，这样可以用旧策略采的样本更新新策略。
>
> 但问题是 ratio 可能很大（比如 5 或 10），导致策略更新幅度过大。PPO-clip 就是解决这个问题：
> ```
> L = -min(ratio * A, clip(ratio, 1-ε, 1+ε) * A)
> ```
>
> 当 advantage > 0（该回答好）：ratio 被截断到 1+ε，防止过度强化
> 当 advantage < 0（该回答差）：ratio 被截断到 1-ε，防止过度惩罚
>
> 核心思想是**信赖域**：只在旧策略附近做更新，保证训练稳定。"

### Q6: KL 散度的作用？为什么要自适应？

**回答**：
> "KL 散度惩罚新策略偏离参考模型太多。没有 KL 惩罚，模型会 reward hacking（比如重复高奖励短语）。
>
> 固定 kl_coef 的问题：训练初期策略变化小，固定惩罚太重抑制探索；训练后期策略偏移大，固定惩罚又太轻。
>
> 我用了**自适应 KL（Lagrangian style）**：
> ```python
> kl_target = 0.01
> if kl_div > kl_target * 1.5:
>     kl_coef *= 1.5   # KL 太大，加大惩罚
> elif kl_div < kl_target * 0.5:
>     kl_coef /= 1.5   # KL 太小，放松探索
> ```
>
> 这来自 OpenAI 的 PPO 论文变体。实践中 KL 稳定在 0.01-0.02 附近，训练更平滑。"
